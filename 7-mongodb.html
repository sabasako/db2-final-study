PART 1: Document Databases & MongoDB Data Model (7.1)

1.1 Review of the Four Data Models

Before diving into MongoDB, it is essential to understand where it sits among the four database paradigms covered in this course. Each model has a fundamentally different way of organizing and accessing data.

Data Model Core Concept Example Systems Key Characteristic
Relational Data in tables (relations) with rows and columns. Relations linked by foreign keys. PostgreSQL, MySQL, Oracle Normalized, strict schema, SQL, strong ACID
Column-Family Data organized by rows with dynamic columns. Rows grouped into column families. ScyllaDB, Apache Cassandra, HBase Query-driven design, wide rows, P2P distributed
Key-Value Simple pairs: key maps to a value. Only 3 operations: SET/PUT, GET, DELETE. Redis, DynamoDB Extreme simplicity, very high performance, no query by value
Document Key maps to a structured document (JSON/BSON/XML). Documents queried by field tags. MongoDB, CouchDB Flexible schema, self-contained documents, rich queries
Mini-World driven vs. Query-Driven Design: In a relational DB, you model the real-world (entities, relationships) and then normalize. In ScyllaDB (column-family), you design tables around query patterns first. MongoDB takes a middle path — you model self-contained, document-like objects that reflect the real world, but you also consider how they will be queried to decide what to embed versus reference.
The slide on data model review asks you to fill in descriptions of all four models. Know them! Also know which databases belong to which model, and what "query-driven design" means for column-family databases compared to MongoDB's approach.

1.2 What is a Document Database?

A document database is an evolution of the key-value concept. In a pure key-value store, the value is opaque — the database cannot see inside it, so queries must always use the key. You cannot say "find all users where email = 'x@y.com'" because the database does not understand the value structure.

Document databases solve this by making the value a tagged document — a structured format like JSON, BSON, or XML where each piece of data has a name (a "tag" or "field"). Because the database understands the structure, it can query by any field, not just the key.

Document Database Definition: A NoSQL database that stores data as key-document pairs. The value part is always a document in a tagged markup format (JSON, BSON, XML). Documents in the same collection can have different fields (flexible/dynamic schema). Related documents are grouped into collections.

Key Properties of Document Databases

  • Queryable by tags/fields: You can filter documents by any field value, enabling sophisticated queries. This is the critical advantage over pure key-value stores.
  • Flexible (dynamic) schema: There is no predefined schema. Each document can introduce new fields. One could also call it "schema-on-read." Schema validation rules can be added optionally at a later stage.
  • Application responsibility: The responsibility for data structure belongs to the application, not the database schema.
  • Self-contained documents: Unlike RDBMS where data is split across multiple normalized tables, a document database document is self-contained and independent in terms of information content.
  • No inherent relationships: Documents are not connected to each other via formal FK-PK constraints. Relationships must be handled by the application (via embedding or manual references).
In a relational DB, data is distributed across different relations to avoid anomalies and to map relationships. In a document DB, each document is self-contained. This means joins are either unnecessary (because related data is embedded) or handled manually by the application (two separate queries).

Why is the tagged document better than a plain key-value for reading?

With a key-value store like Redis, GET user:1234:email gives you the email, but you cannot say "give me all users from Georgia" because the database cannot inspect values. With a document database, you can run: db.users.find({ country: "Georgia" }) because the document's fields are visible and indexed by the database engine. This is the fundamental advantage.

1.3 MongoDB Documents: BSON Structure

MongoDB stores data as BSON (Binary JSON) documents. BSON is a binary-encoded serialization of JSON-like documents. It is faster to parse than text JSON and supports additional data types like ObjectId, Date, and BinData.

The exam may ask: "Which structural elements would not be possible in a normalized relational table?" Answer: arrays, embedded sub-documents, varying fields across rows. 1.4 Embedding vs. Referencing Documents

This is one of the most important design decisions in MongoDB. Unlike a relational database where you always normalize first, MongoDB gives you a choice: embed the related data inside the parent document, or store it in a separate collection and reference it.

Embedded Documents

Embedded Documents are stored inside a parent document. Whenever a parent document is retrieved, all its embedded documents are also retrieved automatically in the same read operation. There is no separate query needed.

When to embed:

  • When data is always accessed together (high locality)
  • 1:1 and 1:N (small N) relationships
  • When the embedded data is not independently queried
  • When the embedded data does not grow without bound
// Student with embedded address — good embedding example
{
  _id: ObjectId("6415523d0ed201caee1a9e5e"),
  userName: 'Wolf',
  firstName: 'Tom',
  lastName: 'Salcher',
  DoB: '2013-02-20',
  gender: 'm',
  mail: 'tom.salcher@postbox.xx',
  lessonBalance: 0
  // address embedded directly — if always needed with student, this is fine
}

Referenced Documents

Referenced Documents are stored in a separate collection. They reference each other using the $ref notation (or manually storing an ID). This reference is ONLY notational — MongoDB does NOT enforce referential integrity like a foreign key. There is no automatic join; you must run TWO separate queries.

When to reference:

  • When the embedded data is large
  • When the embedded data is independently queried (queried on its own, without the parent)
  • When the embedded data is update-and-write heavy (frequent updates require rewriting the whole parent)
  • When there is a 1:N relationship where N is very large (unbounded array anti-pattern)
  • When the same data is shared among multiple parent documents (N:M relationship)
// Student referencing teacher — two-collection approach
// Student document:
{
  _id: ObjectId("6415523d0ed201caee1a9e5e"),
  userName: 'Wolf',
  firstName: 'Tom',
  lastName: 'Salcher',
  DoB: '2013-02-20',
  gender: 'm',
  mail: 'tom.salcher@postbox.xx',
  lessonBalance: 0,
  teachers: {
    "$ref" : "teacher",
    "$id" : ObjectId("5126bc054aed4daf9e2ab772")
  }
}

// To get the teacher, you need a SECOND query:
// 1. db.student.findOne({ userName: 'Wolf' })  → get the teacher ObjectId
// 2. db.teacher.findOne({ _id: ObjectId("5126bc...") })  → get teacher data
MongoDB does NOT support the JOIN concept of FK-PK links. The $ref reference is only notational — it is just a convention for storing the reference. The database itself does NOT enforce it. If you delete the teacher document, the reference in the student document becomes a dangling pointer — MongoDB will not warn you.

1.5 Limitations of the Embedding Concept

Embedding is the most natural and often preferred choice for MongoDB because it gives excellent read performance through data locality (everything in one document = one disk read). However, there are important limitations:

MongoDB typically needs to load the entire document, even if only a small portion of it is accessed. On updates to a document, the entire document usually needs to be rewritten. Therefore, in regard to performance, it is better to keep documents small.
Problem Scenario Why It's a Problem Solution
Large number of embedded sub-documents Document grows large → full document loaded on every read Separate parent and embedded documents into two collections
Embedded documents have many fields themselves Same as above — large document Separate into two collections
Embedded documents are update-and-write heavy Every update to an embedded doc rewrites the entire parent document Separate into two collections for independent writes
Regular queries on embedded documents only Need to retrieve and filter through the entire parent to get embedded data Separate into two collections for query flexibility
Unbounded arrays (arrays growing forever) Document hits the 16 MB size limit; poor performance Reference model — separate collection for the growing data
The professor asks this question in both lecture style: "When should you embed vs. reference?" The rule of thumb is: embed when data is always accessed together and the relationship is 1:1 or 1:few. Reference when data is large, write-heavy, independently queried, or the "many" side can grow unboundedly.

1.6 Use Cases for Document Databases

Document databases are a good fit when the business idea or application:

  • Can be well represented with document-like, self-contained objects
  • Relies mainly on 1:1 or 1:N relationships (not heavy N:M)
  • Does not need many joins (support for joins is weak in document DBs)
  • Profits from better locality (all relevant information is ideally in one document)
  • Needs schema flexibility (different documents can have different fields)
Concrete Use Cases: Catalog applications (products with varying attributes), Content Management Systems (CMS), web applications, user profile systems, event logging, mobile app backends.

PART 2: MongoDB Replication, Consistency & Failover

2.1 Single-Leader Replication

MongoDB uses Single-Leader Replication (also called Primary-Secondary replication). This is a fundamentally different architecture from the Peer-to-Peer (P2P) replication used by ScyllaDB/Cassandra.

Single-Leader Replication Principle:
  • All write requests go to the primary only. The primary processes writes and determines the order in which they are applied.
  • The primary sends a change log (oplog) to the secondaries — in the order writes were processed.
  • Secondaries apply the changes asynchronously — in the order they received them from the primary.
  • Read operations can go to the primary OR a secondary (controlled by Read Preference).
  • A read operation reads from one node only (primary OR one secondary), never from multiple nodes simultaneously.
all write requests | v [ PRIMARY ] / \ oplog oplog / \ [ Sec1 ] [ Sec2 ] - Writes: PRIMARY only - Reads: PRIMARY or one SECONDARY (depending on Read Preference) - Secondaries apply oplog ASYNCHRONOUSLY Chained replication also possible: PRIMARY --> Sec1 --> Sec2 (Sec2 replicates from Sec1, not directly from Primary)
Contrast with P2P (ScyllaDB): In P2P distributed databases, a write goes directly to ALL replicas (based on the consistency level/quorum). A read operation reads from as many replicas as the consistency level requires. There is no single "primary" — any node can accept writes. This is fundamentally different from MongoDB's single-leader model.

Systems that use Single-Leader replication: Oracle, PostgreSQL, MySQL, MongoDB, CouchBase, Neo4j, and in certain configurations: Redis.

2.2 MongoDB Replica Sets

Replica Set: In MongoDB, replication is implemented using a Replica Set (the rs object). A replica set is a set of mongod server instances that store the same data. It contains:
  • Primary node: One primary, handles ALL write operations.
  • Secondary nodes: One or more secondaries that replicate from primary.
  • Replicas are installed typically on separate servers using the same port, but can run on one system using different ports (for testing).

Initializing a Replica Set in mongosh

// Connect to the primary node and run:
rs.initiate({
  _id: "rs1",
  members: [
    { _id: 0, host: "localhost:27018" },
    { _id: 1, host: "localhost:27019" },
    { _id: 2, host: "localhost:27020" }
  ]
})
// Response: { ok: 1 }

// Check replica set status:
rs.status()       // Full comprehensive information on all nodes and their status
rs.status().ok    // Returns 1 if the current node is ok

// The prompt shows whether you are on primary or secondary:
// rs1 [direct: primary] test>    <-- you are on primary
// rs1 [direct: secondary] test>  <-- you are on secondary
The prompt in mongosh tells you your current role. rs.status() returns comprehensive information. rs.status().ok returns 1 for the current mongod instance status. Know the difference.

2.3 The Oplog (Operations Log)

Oplog (Operations Log): A special capped collection stored on the primary node at local.oplog.rs. It records every write operation that modifies data on the primary. Secondaries "tail" the oplog — they continuously read new entries from it and apply those operations to their own data.

Key Properties of the Oplog

  • Capped collection: Has a fixed maximum size. Old entries are overwritten when the size limit is reached.
  • Ordered: Operations are recorded in the exact order the primary processed them.
  • Idempotent: Every operation in the oplog is idempotent — applying it once or multiple times produces the same result. This is critical for reliability.
Idempotent: f(x) = f(f(x)). A function applied multiple times gives the same result as applying it once.

The MongoDB oplog converts non-idempotent write commands into idempotent ones before storing them. For example, $inc: {price: 10} is NOT idempotent — applied twice would add 20 instead of 10. The oplog would instead store the operation as a SET to the final value: $set: {price: <final_value>}. This way, replaying the oplog multiple times is safe.

Why does the oplog need idempotency?

When a secondary restarts after a crash or falls behind, it replays oplog entries it may have already applied. If those operations were not idempotent, replaying them would corrupt the data. With idempotent oplog entries, replaying is always safe — the result is the same regardless of how many times the entry is applied.

2.4 MongoDB Write Concern

Write Concern defines what an "acknowledgment: true" returned to the client actually means — that is, how many nodes need to confirm the write before it is reported as successful. Write concern controls the durability of writes and allows the application to trade off performance vs. durability.
Write Concern Meaning Rollback Possible? Performance
w: 0 Fire and forget — write operation is NOT acknowledged. Client does not wait for any confirmation. Yes (no confirmation of any kind) Highest (no waiting)
w: 1 Acknowledged by the primary only. Write is confirmed when successfully completed on the primary node. Yes (if primary fails before replicating) High
w: 2 Acknowledged when write completed on primary AND one secondary. Rollback still possible (if that one secondary also fails) Medium
w: majority Default. Acknowledged when write completed on majority of nodes. E.g., in a 3-node set: primary + 1 secondary must confirm. Not possible once acknowledged Lower (waits for majority)
w: N Acknowledged when write completed on exactly N nodes. In a 3-node set, w:3 = all nodes must confirm. Depends on N vs total nodes Lowest for w = total nodes
Write concern allows the application to prioritize performance OR durability. With w:majority, a rollback is NOT possible after the acknowledgment is returned. This is because majority-committed writes become the "durable" baseline — even if the old primary returns, it cannot have more writes acknowledged by a majority.

2.5 MongoDB Read Concern

Read Concern controls the consistency level of a read operation. It determines what version of data a read operation returns. WriteConcern and ReadConcern are independent of each other — you can combine them freely.
Read Concern Behavior Stale Read? Dirty Read? Notes
local Returns the most recent data on the node being read. No consistency guarantees. Possible (secondary may be behind) Possible (data may be rolled back) Default. Fastest. "May be stale, even dirty read"
majority Returns data acknowledged by majority of replica set. No dirty reads. Still possible (returns the majority snapshot, which may not be the latest) No dirty reads More consistent but slightly slower
linearizable Only for reads on the PRIMARY. Only for single document reads. Strongest guarantee. No No Slowest. Reflects all majority-committed writes before the read starts.
Application guidance (from slides):
  • If the application can tolerate a read that may not be durable: use readConcern: local
  • If the application CANNOT tolerate a read that may not be durable: use readConcern: majority

Understanding the Read Concern Timeline (from slide p. 26-27)

Consider a 3-node replica set (Primary, Secondary1, Secondary2) with w: majority. A write happens at time t0 that changes data object x from 100 to 200. The acknowledgment is sent back at t6 (after majority = Secondary1 confirms at ~t3-t5).

Write: x=100 → x=200 at time t0 with w:"majority" Timeline: t0 t1 t2 t3 t4 t5 t6 |---write--| | | | | | Primary \ /\ / | \ / \ / ↑ t1-t2 X X Acknowledge sent to client at t6 / \ / \ / \/ \ t1 t4-t5 t2 Sec1 gets at t1, Sec2 gets at t2, acks flow back ReadConcern LOCAL: - Read on Primary after t0 → returns 200 (primary already wrote it) - Read on Secondary1 before t1 → returns 100 (hasn't got the update yet) - Read on Secondary1 after t1 → returns 200 (it applied the oplog) - Read on Secondary2 before t2 → returns 100 (still stale) - Read on Secondary2 after t2 → returns 200 ReadConcern MAJORITY: - Read on Primary before t3 → returns 100 (majority not yet confirmed → reads old) - Read on Primary after t3 → returns 200 (majority confirmed) - Read on Secondary1 before t5 → returns 100 - Read on Secondary1 after t5 → returns 200 - Read on Secondary2 before t6 → returns 100 - Read on Secondary2 after t6 → returns 200
MongoDB write concern/read concern and Cassandra/ScyllaDB consistency levels are strongly related concepts — both control replica participation and determine visibility. However, they differ because MongoDB's primary-based replication is ordered and does not need conflict resolution. Primary-based replication may also discard writes (rollback), while P2P uses conflict-free resolution strategies.

2.6 Read Preference (Read Routing)

Read Preference determines WHICH NODE (server) a read request is routed to. It does not control the consistency level (that is Read Concern's job). Read Preference can be set as a query option, a connection option, or a driver option.
Read Preference Mode Behavior Use Case
primary Default. All reads go to the primary node only. Strong consistency needed — no stale reads possible from routing.
primaryPreferred Reads go to primary if available; falls back to secondary if primary is unavailable. High availability — can tolerate short primary downtime
secondary All reads go to a secondary. Potentially stale data. Offload read traffic from primary; analytics on secondary
secondaryPreferred Reads go to secondary if available; falls back to primary. Read scaling — most reads go to secondaries
nearest Routes to the node with the lowest latency (could be primary or any secondary). Multi-datacenter — read from nearest geographic node. A secondary in DC2 with priority:0 handles nearest reads.
Read Preference "nearest" with multiple data centers: In the slide example, Data Center 1 has a Primary (priority:1) and a Secondary (priority:1). Data Center 2 has only a Secondary (priority:0 — cannot become primary). With Read Preference = nearest, reads from clients in Data Center 2 go to the local Secondary (lowest latency), even though that secondary may have slightly stale data.
Read Preference is DIFFERENT from Read Concern. Read Preference = which node to read from. Read Concern = what version of data to return. You can set Read Preference = secondary and Read Concern = majority, but note that Read Concern majority on a secondary means it will wait for the majority-acknowledged snapshot on that secondary — which may be older than what the primary has. These concepts work together but are independent.

2.7 Replication Lag and Stale Reads

Replication Lag: Because secondaries apply the oplog asynchronously, there is a delay between when the primary processes a write and when that write is reflected on a secondary. This delay is called the replication lag.

Consequences of replication lag:

  • Read requests to secondaries may return stale data (old values not yet updated)
  • The more secondaries lag behind, the more likely reads return stale data

Classic Problem: Read-Your-Own-Write Anomaly (from Slides p. 29)

User 1234 scenario: 1. User 1234 inserts a comment: INSERT INTO comments (author, reply_to, message) VALUES (1234, 55555, 'Sounds good!') 2. Leader processes insert → sends "insert ok" to User 1234 3. User 1234 immediately queries: SELECT * FROM comments WHERE reply_to = 55555 → The query is routed to Follower 2, which hasn't applied the oplog yet → Result: "no results!" — the user doesn't see their own just-written comment! This is the "read-your-own-write" problem caused by replication lag.

Common reasons secondaries fall behind:

  • Network congestion between primary and secondary
  • Secondary is under heavy read load and cannot keep up with applying oplog
  • Secondary hardware is slower than primary
  • Long-running operations on secondary blocking oplog application

Measures to decrease replication lag (that you can control):

  • Use w: majority write concern — the client only gets acknowledgment after majority confirms, so reads from those secondaries are fresher
  • Route reads to primary (Read Preference: primary) to avoid lag entirely
  • Reduce write load on secondary (fewer queries routed there)
  • Keep documents smaller (faster to apply oplog entries)
  • Avoid large batch writes that create many oplog entries at once

2.8 Failover Strategy

Failover is needed for Primary-Secondary (single-leader) distributed databases — NOT for P2P. Reason: In P2P, any available node can accept writes, so there is no single point of failure that requires a failover procedure. In MongoDB, if the primary fails, writes are impossible until a new primary is elected.
Native PostgreSQL does NOT come with automatic failover. MongoDB DOES come with built-in automatic failover through the replica set election mechanism.

A failover system needs:

  1. Failure detection: Ability to detect that the primary has failed or become inaccessible.
  2. Automatic failover: Ability to automatically elect a new primary from the remaining secondaries and resume normal operation.

2.9 Failure Detection and Election Process

Failure Detection: Heartbeats

MongoDB uses heartbeats (pings) for failure detection.
From MongoDB docs: "Replica set members send heartbeats (pings) to each other every two seconds. If a heartbeat does not return within 10 seconds (heartbeatTimeout), the other members mark the delinquent member as inaccessible."

This timeout can be tuned with the settings.heartbeatTimeoutSecs parameter.

Desired properties of failure detection:

  1. Completeness: All running nodes eventually notice when a member fails.
  2. Efficiency: The algorithm detects failure fast (within seconds).
  3. Accuracy: The algorithm should not falsely declare a running (but delayed) node as dead.
Increasing heartbeatTimeout (e.g., 10s → 12s):
  • It takes longer to declare a node dead → increases accuracy (fewer false positives — a slow but running node is less likely to be declared dead)
  • But it takes longer to detect a real failure → decreases efficiency
Answer from the slide: Increasing the timeout increases accuracy (Answer B).

Election Process

MongoDB Automatic Failover Election:
  1. When a primary does not communicate with other members anymore (heartbeat timeout), eligible secondaries detect the failure.
  2. An eligible secondary calls for an election.
  3. An election can be held only if the majority of voting nodes is available.
  4. The election process is based on Raft algorithm concepts.
  5. As soon as a new primary is elected, normal operations resume.
  6. No write processing occurs during the election.

2.10 Split Brain Problem

The split brain problem occurs when a network partition divides a replica set into two groups that cannot communicate with each other. The key question is: can two primaries exist simultaneously?

Scenario 1: Primary in Europe, 3 Secondaries in US (network fault between them)

Datacenter US: [Network Fault] Datacenter Europe: [Secondary] [Primary] [Secondary] ~~~ partition ~~~ [Secondary] [Secondary] US side has 3 nodes, Europe side has 2 nodes (Primary + Secondary). Total replica set = 5 nodes. Majority = 3. Europe side (2 nodes): Europe cannot reach US. Does Europe have majority? → 2 out of 5 = NO majority. Europe primary STEPS DOWN. No writes accepted. US side (3 nodes): US secondaries cannot reach primary in Europe. → US has 3 nodes = majority of 5. US calls an election. → New primary elected in US. Writes resume in US. Result: The old Europe primary steps down. US elects new primary. No split brain possible — only one side has majority. Can there be data loss (rollback)? YES — any writes the Europe primary accepted before stepping down that were NOT replicated to the US majority will be ROLLED BACK.

Scenario 2: Primary in US, 3 Secondaries in Europe (network fault)

Datacenter US: [Network Fault] Datacenter Europe: [Primary] [Secondary] [Secondary] ~~~ partition ~~~ [Secondary] [Secondary] US side: Primary + 2 secondaries = 3 nodes = majority of 5. Primary stays as primary. Europe side: 2 secondaries only. Cannot reach primary. → Try to elect new primary? NO — only 2 nodes, not majority. → Europe secondaries just wait/become read-only. Result: US primary continues. No split brain. Europe becomes read-only. Can there be data loss? Only if US primary fails before network heals.

Heartbeat Delayed Scenario (false positive)

Datacenter US (3 secondaries) ← heartbeat delayed ← Datacenter Europe (Primary + 1 Secondary) US secondaries think primary is DOWN (heartbeat didn't come within 10s timeout). But primary is UP — heartbeat was just slow. US side: 3 nodes = majority. US calls election. Elects new US primary. Europe side: Real primary is still running! Now we have TWO primaries briefly. → MongoDB handles this via the election epoch/term mechanism (Raft): The new US primary has a higher term number. The old Europe primary, when it can communicate again, sees the higher term and STEPS DOWN. It becomes secondary. → Data written to old Europe primary during this window gets ROLLED BACK. Split brain CAN temporarily occur, but is resolved quickly by the Raft term mechanism.

2.11 Rollback in MongoDB

MongoDB Rollback Definition: "A rollback reverts write operations on a former primary when the member rejoins its replica set after a failover. A rollback is necessary only if the primary had accepted write operations that the secondaries had NOT successfully replicated before the primary stepped down. When the primary rejoins the set as a secondary, it reverts, or 'rolls back,' its write operations to maintain database consistency with the other members."

Fundamental Difference: MongoDB Rollback vs. Postgres Rollback

Aspect MongoDB Rollback PostgreSQL Rollback
Cause Happens AUTOMATICALLY due to a failover/election — after the old primary rejoins as secondary. It is a REPLICATION phenomenon. Happens INTENTIONALLY — a client explicitly issues ROLLBACK (or a transaction fails). It is a TRANSACTION phenomenon.
Who triggers it The MongoDB system automatically — the old primary triggers its own rollback when it rejoins and sees it has diverged from the new majority. The application or database engine (on error) — it is always triggered by a transaction ABORT command.
What is lost Writes that were acknowledged to the client (with w:1) but not replicated to majority before the failover. These writes are PERMANENTLY LOST (written to a rollback file, but not automatically re-applied). Changes within an open transaction that was not committed. The transaction was never visible to other transactions (depending on isolation level).

This comparison (MongoDB rollback vs. Postgres rollback) is an explicit assignment question. The key answer: MongoDB rollback is a REPLICATION phenomenon caused by failover — it can lose acknowledged data (with w:1). Postgres rollback is a TRANSACTION concept — it is intentional and does not lose committed data. MongoDB rollback can be prevented by using w:majority.

PART 3: MongoDB Sharding / Partitioning (7.2)

3.1 Why Sharding?

Sharding (Partitioning): The data is split up into subsets. These subsets are called "partitions" in P2P distributed databases, and they are called shards in MongoDB. Each shard stores a portion of the total dataset.

A single replica set can only handle so much data and throughput. When you reach that limit, you need to scale horizontally — spread the data across multiple replica sets. This is sharding.

Key facts from the slides:

  • Nodes (shards) are physical or virtual servers.
  • Each shard MUST be deployed as a replica set — you get both horizontal scaling AND redundancy.
  • Even though replication and sharding are different concepts, they are typically combined together.
SHARDING + REPLICATION COMBINED (e.g., from slide "Sharding" diagram): Shard 1 (A-F range) Shard 2 (G-P range) Shard 3 (Q-Z range) ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ │ Master DB │ │ Master DB │ │ Master DB │ │ Read-only slave │ │ Read-only slave │ │ Read-only slave │ │ Read-only slave │ │ Read-only slave │ │ Read-only slave │ └─────────────────┘ └─────────────────┘ └─────────────────┘ ↑ ↑ ↑ |____________________|_____________________| ↑ Memcached servers ↑ Web servers (3x) Each shard is itself a replica set (master + 2 read-only slaves). Sharding handles the PARTITION of data. Replication handles REDUNDANCY within each shard.

3.2 Sharded Cluster Architecture

A MongoDB sharded cluster has three types of components:

1. Shards: Each shard is a replica set that stores a subset of the data. In the diagram, Shard1, Shard2, ... ShardN, each with Primary + Secondary + Secondary.

2. Mongos (Router): The client connects to mongos, NOT directly to shards. Mongos is a query router that knows (from the config server) which shard holds which data. It routes client queries to the correct shard(s). Multiple mongos routers can exist for high availability.

3. Config Server: Stores the configuration settings of the whole cluster — specifically the chunk map (which shard key range lives on which shard). Must ALSO be deployed as a replica set. Mongos caches the chunk map from the config server.

Config Server = replica set storing metadata (chunk distribution). Mongos = stateless router; caches chunk map from config server.

3.3 Shard Key Strategies

Shard Key: The field (or set of fields) used to distribute documents across shards. The shard key controls the distribution of data. It is one of the most critical decisions in a sharded MongoDB deployment.

MongoDB divides the shard key's value space into chunks. Each chunk is assigned to a shard. When you insert a document, MongoDB hashes or evaluates the shard key value and routes the document to the appropriate chunk (and therefore shard).

Strategy 1: Range Key (Range Sharding)

Range Sharding: The shard key partitions the data into ranges. Each shard holds a contiguous range of shard key values.

Example: Shard key = postalCode, ranges: 10000–30000, 30000–60000, 60000–99999
Each shard holds all documents with postal codes in its range. 
// Range sharding visual (from slide p.8 left side):
Document { X: 25 } → falls in range [10 ≤ X < 20]? No. Falls in [20 ≤ X < maxKey]? Yes → Chunk C → Shard 3
Document { X: 26 } → falls in [20 ≤ X < maxKey]? Yes → Chunk C → Shard 3
Document { X: 27 } → falls in [20 ≤ X < maxKey]? Yes → Chunk C → Shard 3

// Problem: all three documents go to the SAME shard. This is a "hot spot".

Advantages of Range Sharding:

  • Range queries are highly efficient — all documents in a range are on the same shard (or a few shards). The mongos can route the query to just one or a few shards (targeted query).
  • Useful for applications that frequently retrieve sets of items using range queries (e.g., "get all orders from January to March").
  • Groups related items together on the same shard.

Disadvantages of Range Sharding:

  • Hot spots: If the shard key is monotonically increasing (like a timestamp or auto-increment ID), all new insertions always go to the last chunk (highest range) on one shard. That shard becomes a bottleneck while others sit idle.
  • Uneven distribution if the data is not uniformly distributed across the key range.

Strategy 2: Hashed Key (Hash Sharding)

Hash Sharding: The shard key value is hashed by a hash function. The hash value determines which chunk (and shard) the document goes to. The hash distributes documents uniformly across shards regardless of the shard key's natural ordering. 
// Hash sharding visual (from slide p.8 right side):
Document { X: 25 } → hash(25) = some_hash_value → could map to Chunk A, B, or C randomly
Document { X: 26 } → hash(26) = different_hash → maps to a different chunk
Document { X: 27 } → hash(27) = yet another → maps to yet another chunk

// Even though 25, 26, 27 are sequential, they land on DIFFERENT shards → no hot spot!

Advantages of Hash Sharding:

  • Reduces the chance of hot spots — distributes data across nodes evenly even with sequential shard keys.
  • Good for high-throughput insert workloads.

Disadvantages of Hash Sharding:

  • Range queries are expensive (scatter-gather): Documents with similar shard key values are scattered across all shards. A range query like "find all orders from January" must be sent to ALL shards and the results aggregated by mongos — a scatter-gather operation.
  • No data locality for related data.

Shard Key Modifications

The slide (p.10) raises important edge cases about shard key behavior:

If the value of a shard key changes: (e.g., teacher with ID1 moves from postal code 0100 to 0101)
Since the shard key determines which shard the document lives on, changing it means the document must MOVE to a different shard. MongoDB 5.0+ supports shard key value updates, but the operation involves deleting from the old shard and inserting on the new shard. Before MongoDB 5.0, the shard key was immutable.

If the shard key value is missing: Since schema is flexible in MongoDB, a document might not have the shard key field. MongoDB treats a missing shard key value as null, placing the document in the "null" chunk, usually on the first shard. This can create hot spots if many documents lack the shard key.

3.4 Chunks and the Balancer

Chunk: MongoDB divides the shard key's value range into chunks. Each chunk has a minimum and maximum shard key value (e.g., minKey ≤ X < 10, 10 ≤ X < 20, 20 ≤ X < maxKey). Each chunk lives on exactly one shard at any given time.

Chunk lifecycle:

  • Auto-split: When a chunk grows beyond the maximum chunk size (default 128MB), MongoDB automatically splits it into two smaller chunks.
  • Balancer: A background process that runs on the config server. It monitors chunk distribution across shards. When one shard has significantly more chunks than others, the balancer moves chunks to less-loaded shards.
  • Balancing window: In production, balancing is often configured to run during off-peak hours to avoid performance impact.
The chunk/balancer concept connects directly to the shard key strategy. With range sharding and a sequential key, one shard keeps getting new data (hot spot) and the balancer has to keep moving chunks — this is extra overhead and not ideal. Hash sharding avoids this by distributing inserts from the start.

3.5 Targeted vs. Scatter-Gather Queries in Sharding

Targeted Query: A query that includes the shard key in its filter. Mongos can look up the chunk map and route the query to exactly the shard(s) holding the relevant data. Very efficient.

Scatter-Gather Query: A query that does NOT include the shard key. Mongos must send the query to ALL shards and then merge/aggregate the results. Expensive — grows linearly with the number of shards. 
// Shard key = postalCode

// TARGETED query — includes shard key:
db.teachers.find({ postalCode: { $gte: 4000, $lte: 5000 } })
// Mongos routes to only the shard(s) holding postal codes 4000-5000. Fast!

// SCATTER-GATHER query — no shard key:
db.teachers.find({ gender: "f" })
// Mongos sends to ALL shards. Each shard searches locally. Results merged. Slow!

Assignment Block 1: Asynchronous Replication Problems

Context: MongoDB – as other single-leader distributed databases – replicates asynchronously.

Q1: Name at least 2 problems that stem from asynchronous replication and explain the problems.
Problem 1: Stale Reads
Because secondaries apply the oplog asynchronously, there is a delay (replication lag) between when a write is committed on the primary and when it appears on a secondary. If a client reads from a secondary during this lag, it receives old (stale) data — data that does not reflect the most recent committed write. This is called a stale read. Example: A price is updated to 50€ on the primary, but a secondary still shows 45€ because it hasn't processed the oplog entry yet.

Problem 2: Read-Your-Own-Write / Non-Monotonic Reads
A user inserts a document (e.g., a comment) and the primary confirms it. But then the user's next read is routed to a secondary that hasn't yet replicated that document. The user doesn't see their own just-written comment — this is the "read-your-own-write" anomaly. This can confuse users ("I just posted that, where did it go?").

Problem 3: Dirty Reads / Rollback Risk
With readConcern: local (default), a client may read a write from the primary that has NOT yet been replicated to a majority. If the primary fails before that write is replicated, it gets rolled back. The client then has data that "never officially existed" — a dirty read that was subsequently rolled back.

Problem 4: Loss of Data on Failover (Rollback)
If the primary accepts writes with w:1 and then fails before those writes reach a majority of secondaries, when a new primary is elected, those writes are rolled back. Acknowledged writes are permanently lost.
Q2: How can you force MongoDB replication to behave synchronously?
Use Write Concern w:majority. With w:majority, the primary waits until a majority of replica set nodes (e.g., 2 out of 3) have confirmed they received and applied the write before sending acknowledgment to the client. This effectively makes replication synchronous from the client's perspective — the client does not get the "success" response until the write has been propagated to majority. Note: the replication mechanism itself is still technically asynchronous under the hood, but the client is blocked until the majority confirms, making the observable behavior synchronous.

Alternatively, use w:N where N is the total number of nodes to force all nodes to confirm (fully synchronous), at the cost of higher latency.

Assignment Block 2: Replication Lag Diagram Analysis

Q2: Describe the replication lag problem displayed in the picture.
The diagram shows the read-your-own-write anomaly caused by replication lag:
  1. User 1234 inserts a comment. The leader (primary) processes it and sends back "insert ok" immediately (w:1).
  2. The leader begins sending the insert operation to followers asynchronously via the oplog. Follower 1 receives and applies it relatively quickly. But Follower 2 receives and applies it much later (significant replication lag).
  3. After receiving "insert ok", User 1234 immediately queries for their comment (WHERE reply_to = 55555). This read is routed to Follower 2 (e.g., via secondaryPreferred read preference).
  4. Follower 2 has NOT yet applied the insert oplog entry. So the query returns "no results!" — even though the user just successfully inserted the comment seconds ago.
This is the replication lag problem: the secondary is behind the primary, leading to inconsistent reads where a client cannot see data it just wrote.
Q3: What measures can you take to decrease replication lag? (Only list measures you can control — not hardware/network improvements.)
Controllable measures to decrease replication lag:
  1. Use higher write concern (w:majority or w:all): The primary waits for secondaries before acknowledging. This ensures secondaries are up-to-date before the client reads again.
  2. Route reads to primary (readPreference: primary): By reading from primary instead of secondaries, replication lag is irrelevant for reads. The tradeoff is more load on primary.
  3. Reduce the read load on secondaries: If secondaries are overwhelmed with read queries, they fall behind applying the oplog. Reducing reads on secondaries gives them more CPU/IO for oplog application.
  4. Keep documents smaller: Large documents create larger oplog entries. Smaller documents are faster to replicate and apply.
  5. Avoid large bulk operations: Writing many documents in one batch creates a burst of oplog entries that secondaries must process all at once, causing temporary spikes in lag.
  6. Read Preference: primaryPreferred: Combined with application logic to detect when reading recently written data, ensures the primary is used for reads that need freshness.

Assignment Block 3: MongoDB Replication Idempotent OpLog

Context: "Each operation in the oplog is idempotent. That is, oplog operations produce the same results whether applied once or multiple times to the target dataset." idempotent: f(x) = f(f(x))

Q1: In what cases / scenarios do oplog operations need to be applied more than once?
Oplog operations may need to be applied more than once in the following scenarios:
  • Secondary crashes mid-replication: A secondary may crash while applying oplog entries and not record which entries were fully applied. On restart, it may re-apply some oplog entries that were already applied before the crash. Idempotency ensures no harm is done.
  • Network interruption during oplog tailing: A secondary is tailing the primary's oplog. A network hiccup interrupts the connection. When the connection is restored, the secondary may request oplog entries from a slightly earlier point (to be safe) and some entries get applied a second time.
  • Rollback and resync: When a former primary rejoins as secondary after failover, it may need to replay oplog operations to catch up to the new primary. Some entries may overlap.
  • Initial sync retry: During initial synchronization of a new secondary, if the sync is interrupted and restarted, some operations may be applied multiple times.
Q2: Update commands may or may not be idempotent per se. Give an example of an update command that is NOT idempotent. Assume we have a collection of product documents.
A non-idempotent update command using $inc: 
db.product.updateOne(
  { _id: ObjectId("64f123456789abcdef123456") },
  { $inc: { price: 10 } }
)
// Applied once:  price goes from 100 → 110
// Applied twice: price goes from 100 → 120  (NOT the same result!)
// Applied three times: price → 130
// This is NOT idempotent: f(f(x)) ≠ f(x)
Similarly, $mul: { price: 1.2 } (multiply price by 1.2) is not idempotent — applied twice would multiply the price twice.

Also non-idempotent: $push: { items: "new_item" } — applied twice pushes the same item twice into the array.
Q3: What about non-deterministic functions? Give an example.
Non-deterministic functions produce different results each time they are called, even with the same input. Example: using the current timestamp or a random number as part of an update: 
db.product.updateOne(
  { _id: ObjectId("64f123...") },
  { $set: { last_modified: new Date() } }  // new Date() = current time — different each call!
)
// Applied at 10:00 → last_modified = 10:00
// Applied again at 10:05 → last_modified = 10:05  (different result!)
Another example: $set: { discount: Math.random() * 10 } — every application gives a different discount value.

The MongoDB oplog handles this by storing the RESULT of the non-deterministic function, not the function call itself. So the oplog entry would store $set: { last_modified: ISODate("2026-06-22T10:00:00Z") } — the concrete value, not new Date(). This makes the oplog entry deterministic and idempotent.
Q4: What does an oplog update operation entry need to contain in order to be idempotent? Give an example using the product collection.
To be idempotent, an oplog update entry must use a $set operation with the absolute final value, not a relative change. It must store the result state, not the delta.

Example:
Original user command (NOT idempotent): db.product.updateOne({ _id: ... }, { $inc: { price: 10 } })
Oplog stores the idempotent version: { op: "u", ns: "shop.product", o: { $v: 2, diff: { u: { price: 110 } } }, o2: { _id: ObjectId("...") } }

In plain terms: the oplog records "SET price TO 110" (the final absolute value), not "ADD 10 TO price". If this is applied once or multiple times, price always ends up at 110. This is the key principle:
  • Original command: { $inc: { price: 10 } } → NOT idempotent
  • Oplog transformation: { $set: { price: 110 } } → IS idempotent
The oplog must contain:
  1. The operation type (insert, update, delete)
  2. The exact document _id being modified (so repeated application hits the same document)
  3. An absolute $set of the final field values (not relative changes like $inc, $mul, $push)

Assignment Block 4: Read Preference = Primary — Stale and Dirty Reads

Context: Assume that you set the MongoDB Read Preference to Primary.

Q1: In this case, can a read request still return stale data? If yes, precisely explain when and why. If no, precisely explain why not.
Answer: No, with Read Preference = Primary, stale reads are not possible from a routing perspective.

When Read Preference is set to Primary, ALL read requests are routed exclusively to the primary node. The primary is the node where ALL writes happen first. It always has the most up-to-date version of the data. Since reads go to the primary and writes also go to the primary, there is no lag between write and read — the primary always has the latest committed data.

Note: "stale" in the context of replication lag means reading from a secondary that is behind the primary. Since we never read from secondaries in this configuration, replication lag cannot cause stale reads.

However, note that with readConcern: local (default), there is a theoretical edge case: the primary may have accepted a write that has not yet been replicated to majority and may be rolled back. The read would return data that is "fresh" (not stale in terms of lag) but may later be rolled back (dirty). See Q2 for this case.
Q2: In this case, can a read request still return dirty reads that may be rolled back? If yes, precisely explain when and why. If no, precisely explain why not.
Answer: YES, with Read Preference = Primary AND readConcern: local (default), dirty reads are still possible.

Here is the scenario:
  1. A write is sent to the primary with write concern w:1.
  2. The primary writes the data and immediately returns "acknowledged: true" to the client.
  3. A read (with readConcern: local, Read Preference: primary) is executed and returns this new data.
  4. BUT: the write has NOT yet been replicated to any secondary (or to majority).
  5. The primary then FAILS before replicating the write.
  6. A new primary is elected. It does NOT have this write (it was only on the old primary).
  7. When the old primary rejoins as secondary, it rolls back this write.
The read in step 3 returned data that was subsequently rolled back — this is a dirty read.

How to prevent it: Use readConcern: majority. With this setting, the primary only returns data that has been acknowledged by the majority of the replica set — data that CANNOT be rolled back. Even though you are reading from the primary, majority-read-concern ensures you only see durable data.

Assignment Block 5: Rollback — MongoDB vs. Postgres

What is the fundamental difference between a rollback in MongoDB and a rollback in Postgres?
MongoDB Rollback:
  • Cause: Happens AUTOMATICALLY due to a PRIMARY FAILOVER in a replica set. It is a replication/distributed systems phenomenon.
  • When: After a failover, the old primary rejoins as secondary. If it had writes that were NOT replicated to the majority before it failed, those writes conflict with the new primary's state. The old primary automatically rolls back those writes to align with the new primary.
  • Data loss: YES — acknowledged writes (with w:1) may be permanently lost. The rolled-back data is written to a special rollback file but is NOT automatically re-applied.
  • Application impact: The application may have already received confirmation of the write and even presented the data to users. That data then disappears.
  • Prevention: Use w:majority. With majority write concern, once acknowledged, a write CANNOT be rolled back because majority means it is on enough nodes that the new primary will also have it.
PostgreSQL Rollback:
  • Cause: Happens INTENTIONALLY when a client issues ROLLBACK, or AUTOMATICALLY when a transaction fails due to an error (constraint violation, deadlock, etc.).
  • When: Within a single transaction that has not yet been COMMITTED. The transaction changes are undone.
  • Data loss: NO — since the transaction was never committed, no externally visible, durable data is lost. Other transactions were never able to see the uncommitted changes (depending on isolation level).
  • Application impact: Clean — the application knows the operation failed and can retry.

Key fundamental difference in one sentence: A Postgres rollback is an intentional transaction abort within a single node that never loses committed data. A MongoDB rollback is an automatic, involuntary loss of already-acknowledged writes triggered by a failover event in a distributed system — it can lose data the client was told was successfully saved.

Assignment Block 6: Split Brain — Scenario 1 (Primary in Europe, 3 Secondaries in US)

Context: 5-node replica set: Datacenter Europe has [Primary + 1 Secondary]. Datacenter US has [3 Secondaries]. Network fault cuts connection between US and Europe.

What happens when the network connection breaks? How do Europe nodes react? How do US nodes react? Can there be a split brain? Can there be data loss (rollback)?
Setup: 5-node replica set total. Europe: Primary (1) + Secondary (1) = 2 nodes. US: 3 Secondaries = 3 nodes. Majority of 5 = 3.

Europe nodes react:
  • Europe has 2 nodes. They can still communicate with each other.
  • Europe's primary can no longer reach the 3 US secondaries.
  • The primary checks: do I still have a quorum (majority)? 2 out of 5 total = NOT a majority. The primary STEPS DOWN voluntarily. It can no longer accept writes.
  • The Europe secondary also cannot become primary (only 2/5 nodes available = not majority).
  • Europe nodes become read-only (no primary, no writes accepted). Note: with readPreference:secondaryPreferred, reads can still be served from Europe's secondary.
US nodes react:
  • US has 3 secondaries. They can communicate with each other but not with Europe.
  • US sees that the primary (in Europe) is unreachable — heartbeat timeout exceeded.
  • US calls an election. They have 3 votes = majority of 5. An election is valid.
  • One of the US secondaries is elected as the new primary.
  • Writes resume on the new US primary. Normal operations continue in the US datacenter.
Can there be a split brain? No. A split brain would mean two primaries exist simultaneously. However, because the majority rule is enforced: the Europe side (2 nodes) cannot elect a primary (not majority), so it steps down. Only the US side (3 nodes = majority) can hold an election. At no point do two primaries coexist.

Can there be data loss (rollback)? YES, there can be data loss. Any writes that the Europe primary accepted (with w:1) BEFORE it stepped down and that were NOT yet replicated to any of the US secondaries will be rolled back when the network heals and the Europe primary rejoins as secondary. The old Europe primary will revert those writes. With w:majority, such a rollback would not be possible since majority (US nodes) would have had to confirm.

Assignment Block 7: Split Brain — Scenario 2 (Heartbeat Delayed — "False Positive" scenario)

Context: 5-node replica set: Datacenter Europe has [Primary + 1 Secondary]. Datacenter US has [3 Secondaries]. Heartbeat from primary (Europe) is delayed to US — US thinks primary is down but primary is actually up.

What happens? How do Europe nodes react? How do US nodes react? Can there be a split brain? Can there be data loss (rollback)?
US nodes react:
  • The 3 US secondaries do not receive heartbeat from the Europe primary within the heartbeatTimeout (10 seconds by default).
  • US concludes the primary is dead and calls an election.
  • US has 3 votes = majority of 5. Election is valid. A new primary is elected in the US.
  • US starts accepting writes on the new primary.
Europe nodes react:
  • The old Europe primary is still UP and running. It still believes it is primary.
  • The Europe primary continues accepting writes from clients who connect to it.
  • The Europe secondary continues replicating from the Europe primary.
Can there be a split brain? YES — temporarily there are TWO primaries: the old Europe primary and the newly elected US primary. This is a temporary split-brain condition. However, MongoDB uses the Raft-based election term/epoch mechanism:
  • When the delayed heartbeat finally arrives, or when the network heals, the old Europe primary communicates with the US nodes.
  • The US nodes are now in a higher election term (they held an election).
  • The old Europe primary detects it is in a stale term. It immediately steps down and becomes a secondary.
  • The split brain is resolved. Only the US primary remains.
Can there be data loss (rollback)? YES — during the window when both primaries were active, writes went to both the old Europe primary and the new US primary. When the Europe primary steps down and rejoins as secondary, any writes it accepted that were NOT replicated to the US majority will be ROLLED BACK. Those writes are permanently lost (unless manually recovered from the rollback file). This is why w:majority is critical: with w:majority, the Europe primary would have needed US nodes to confirm — but US nodes were unreachable — so the writes would have timed out (wtimeout) rather than being acknowledged. No acknowledged-then-rolled-back scenario.

Assignment Block 8: MongoDB Sharding — Camera Pictures (Duckey Town Cars)

Context: Store pictures from cameras of Duckey town cars in a MongoDB collection for later analysis. Many pictures → shard the documents. Pictures are small enough to be stored directly in documents.

Q1: Model a camera picture document. Assume that the pictures are small enough to be stored directly in the documents.
{
  _id: ObjectId("64f123456789abcdef123456"),
  camera_id: "CAM-042",             // which camera took this picture
  car_id: "DT-TRUCK-017",           // which car (vehicle ID)
  timestamp: ISODate("2026-06-22T14:35:22Z"),  // when the picture was taken
  location: {                        // GPS coordinates at time of capture
    type: "Point",
    coordinates: [42.2679, 42.7081]  // [longitude, latitude]
  },
  image_data: BinData(0, "..."),     // binary image data (BSON BinData type)
  image_format: "JPEG",
  resolution: { width: 1920, height: 1080 },
  file_size_bytes: 245760,
  tags: [],                          // optional: analysis tags added later
  metadata: {
    exposure: "1/500s",
    iso: 200,
    weather: "clear"
  }
}
Q2: What range shard key would you choose? Explain.
Recommended range shard key: camera_id (or a compound key of { camera_id: 1, timestamp: 1 })

Reasoning:
  • camera_id as range shard key: All pictures from the same camera are stored on the same shard. This supports common analytical queries like "get all pictures from camera CAM-042 in the last week" — these queries are targeted (go to one shard only). The number of cameras (cardinality) determines distribution — if there are many cameras, data is distributed across many shards.
  • Why NOT timestamp alone? Timestamp is monotonically increasing — all new pictures always go to the "latest" chunk, causing a hot spot on one shard. This is the classic bad range shard key. However, timestamp within each camera_id is fine as a secondary sort key.
  • Why NOT image_data? Binary data should not be a shard key.
  • Compound key { camera_id: 1, timestamp: 1 }: Even better — distributes by camera, and within each camera, orders by time. Supports queries like "all pictures from CAM-042 between time A and B" as targeted queries.
Alternative: Hash shard key on camera_id — if the goal is purely uniform distribution (no need for camera-specific range queries), hashing camera_id distributes evenly. But for analysis use cases where you query by camera, range is better.

Final answer: Choose camera_id as the range shard key. It has relatively high cardinality (many cameras), provides data locality for camera-based analytical queries, and avoids hot spots as long as the cameras are relatively uniformly active. Do NOT choose timestamp alone as the shard key because it is monotonically increasing and would create a hot spot.

6.3 Conceptual Exam Tips — What Prof. Knirsch Tests

Document database vs. key-value: The key advantage of document DB over key-value is that the tagged document can be QUERIED by field values — not just by key. This enables sophisticated queries.
Embed vs. Reference — the decision: Embed when: always accessed together, 1:1/1:few, not independently queried. Reference when: data is large, independently queried, update-heavy, or array is unbounded. MongoDB loads the ENTIRE document even for partial access — keep documents small.
Write concern vs. read concern vs. read preference: These are THREE separate independent settings. Write concern = how many nodes confirm the write. Read concern = what version of data to return. Read preference = which node to read from. They are independent and can be combined.
Rollback in MongoDB: A MongoDB rollback is NOT like a Postgres transaction rollback. It is an automatic event caused by failover that can lose ACKNOWLEDGED writes (w:1). Prevention: use w:majority. This is a frequently tested distinction.
Split brain and majority: MongoDB prevents split brain via the majority voting rule — only the partition with more than half the voting members can elect a primary. The minority partition steps down or cannot elect. Know the math: 5-node set, 3 nodes in one partition, 2 in another → the 3-node partition can elect, the 2-node cannot.
Failover needed? YES for single-leader (MongoDB, Postgres). NO for P2P (ScyllaDB). This is a direct exam question from the slide.
Heartbeat timeout accuracy vs. efficiency: Increasing heartbeatTimeout → more time before declaring a node dead → higher ACCURACY (fewer false positives) but lower EFFICIENCY (slower detection). The slide explicitly tests this.
Idempotent oplog: The oplog converts non-idempotent operations ($inc, $mul, $push, new Date()) into idempotent $set operations storing the final absolute value. This is required so replaying the oplog (e.g., after secondary crash) is always safe.
Shard key choice — range vs. hash: Range = good for range queries, bad with sequential keys (hot spot). Hash = uniform distribution, bad for range queries (scatter-gather). Never choose a monotonically increasing field (timestamp, autoincrement) as a RANGE shard key.
$ref is only notational: MongoDB does NOT enforce references. There are no foreign key constraints. If you delete a referenced document, the reference becomes dangling and MongoDB will NOT warn you. This is the application's responsibility.
Read-Your-Own-Write problem: With w:1, a write is confirmed quickly. But reading from a lagging secondary afterwards may not show the written data. Solution: use readPreference:primary for reads that need to see the latest write, or use w:majority + readConcern:majority.
