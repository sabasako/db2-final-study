1. What is a Transaction? — ACID Properties

A transaction is a logical unit of work in a database — a sequence of SQL operations treated as a single atomic operation. Transactions are the mechanism that makes multi-user databases safe.

1.1 Basic Transaction Commands

BEGIN;            -- start a transaction
  UPDATE product SET stock = stock - 1 WHERE product_id = 3;
  INSERT INTO order_item(order_id, product_id) VALUES (99, 3);
COMMIT;           -- make changes permanent

-- OR:
ROLLBACK;         -- undo all changes since BEGIN

In PostgreSQL's autocommit mode, every single SQL statement is automatically wrapped in its own transaction. When you run BEGIN, you start a multi-statement transaction.

1.2 ACID — The Four Properties

ACID is the set of guarantees a DBMS provides for transactions:

A — Atomicity

Either all operations in a transaction succeed and are committed, or none of them are. There is no partial commit. If anything fails, the entire transaction is rolled back.

  • Example: A bank transfer deducting €100 from Account A and adding €100 to Account B. If the deduction succeeds but the addition crashes, atomicity ensures the deduction is also rolled back. No money is lost.
  • Implemented via the undo log (PostgreSQL: old versions in heap) and WAL.

C — Consistency

A transaction brings the database from one consistent state to another. All defined constraints (primary keys, foreign keys, check constraints, uniqueness) must hold before and after every transaction.

  • Example: If the rule is "stock ≥ 0", a transaction that would set stock to −1 violates consistency and must be aborted.
  • The DBMS enforces constraints automatically; the application must also maintain application-level invariants (business logic).

I — Isolation

Concurrent transactions appear to execute in isolation — as if they ran serially, one after another. One transaction's intermediate state is not visible to other transactions.

  • Example: If T1 is in the middle of updating a row, T2 should not see the half-updated row.
  • Isolation is the most complex ACID property — it has levels (Read Committed, Repeatable Read, Serializable) because full isolation is expensive.
  • Implemented via locking (2PL) or multiversion concurrency control (MVCC).

D — Durability

Once a transaction has been committed, its changes are permanent — they survive crashes, power failures, or any system error. The committed data will not be lost.

  • Implemented via the Write-Ahead Log (WAL) in PostgreSQL.
  • The WAL is flushed to disk before the COMMIT returns to the client.
Key Exam Point: Atomicity and Durability are about individual transaction correctness over time (recovery). Consistency is about constraints. Isolation is about concurrent correctness — preventing interference between concurrent transactions.

2. WAL — Write-Ahead Log (Durability in PostgreSQL)

2.1 The Problem WAL Solves

Databases keep data in a Buffer Pool in RAM for performance. A page must be read from disk into RAM before it can be modified. Modified pages (called dirty pages) sit in RAM but may not be written back to disk immediately. If the system crashes before a dirty page is flushed, data is lost.

The WAL rule: Before any dirty page is written to disk, all WAL log records describing that page's changes must first be written to the WAL on disk. This is the "write-ahead" guarantee.

2.2 How WAL Works

WAL = Sequential append-only log of every change made to the database.
Every INSERT, UPDATE, DELETE, COMMIT generates WAL records that are written before the actual data pages are modified on disk.
  • WAL is stored as a series of files in $PGDATA/pg_wal/
  • Each WAL record has a Log Sequence Number (LSN) — a unique, monotonically increasing 64-bit number (e.g., 0/08D74A80)
  • Each record has a prev pointer to the previous LSN — forming a chain
  • Every WAL record specifies: which transaction, which resource manager (Heap, Btree, Transaction), what operation

WAL Record Types (seen via pg_waldump)

rmgr: Heap        desc: INSERT  tx: 3296  lsn: 0/08D74A80
rmgr: Heap        desc: UPDATE  tx: 3297
rmgr: Heap        desc: HOT_UPDATE  tx: 3296  (heap-only-tuple, no index update)
rmgr: Btree       desc: INSERT_LEAF  tx: 3297  (new index entry)
rmgr: Transaction desc: COMMIT  tx: 3296  timestamp: 2025-04-06 09:40:25

2.3 WAL and COMMIT — Durability

When you run COMMIT:

  1. PostgreSQL writes a COMMIT record to the WAL
  2. The WAL buffer is flushed (fsync) to disk
  3. Only then does PostgreSQL return "COMMIT" to the client
  4. The actual data pages (heap, index) may still be in the buffer pool as dirty pages — they will be written later

This means after a crash, PostgreSQL can replay the WAL from the last checkpoint to reconstruct any lost dirty pages. The data is not lost because the WAL is already on disk.

2.4 Reading the WAL with pg_waldump

-- Find current WAL file location:
SELECT pg_current_wal_lsn();

-- Show WAL from a specific LSN:
pg_waldump --start=0/08D74A00 $PGDATA/pg_wal/000000010000000000000008

-- Show WAL for a specific transaction:
pg_waldump --xid=3297 $PGDATA/pg_wal/000000010000000000000008

Example WAL output:

rmgr: Heap   len: 65, tx: 3296, lsn: 0/08D74A80, prev 0/08D74A48,
   desc: HOT_UPDATE old_xmax: 3296, old_off: 52, new_xmax: 0, new_off: 53
rmgr: Transaction len: 34, tx: 3296, lsn: 0/08D75C48,
   desc: COMMIT 2025-04-06 09:40:25
Key point: HOT_UPDATE in the WAL means the update stayed on the same heap page and did not require an index update (non-indexed column changed, or space available on page). A plain UPDATE means a new CTID was assigned and all indexes were updated.

2.5 Checkpoints

A checkpoint is a point in time where PostgreSQL guarantees that all dirty buffer pool pages have been flushed to disk. After a checkpoint, crash recovery only needs to replay WAL from the checkpoint forward (not from the beginning).

  • Checkpoints happen periodically (checkpoint_timeout, default 5 min) or when WAL fills (max_wal_size)
  • A checkpoint record is written to WAL marking the checkpoint LSN
  • After a crash, recovery starts from the last checkpoint LSN
Summary: WAL guarantees Durability (D in ACID)
  • WAL is written before data pages → survives crashes
  • COMMIT waits for WAL fsync → committed data is permanent
  • Checkpoints limit recovery time → only replay WAL since last checkpoint
  • WAL also enables replication (streaming WAL to standby servers)

3. Buffer Pool, Pages and the Storage Layer

3.1 The Buffer Pool

The PostgreSQL buffer pool is a region of shared memory that caches disk pages. All reads and writes go through the buffer pool:

  1. To read a row, PostgreSQL finds which heap page it's on, reads that page into the buffer pool (if not already there), then extracts the row
  2. To write a row, PostgreSQL modifies the page in the buffer pool (the page is now "dirty")
  3. Eventually, a background process flushes dirty pages to disk

3.2 Heap Pages and Index Pages

PostgreSQL stores data in fixed-size 8KB pages:

  • Heap pages — store the actual rows (tuples) of a table
  • Btree leaf pages — store index entries pointing to heap tuples via (page_number, slot) = ctid

ctid — Physical Row Address

ctid is a PostgreSQL system column containing the physical location of a tuple: (page_number, slot_number). For example, (0,1) means page 0, slot 1.

SELECT ctid, xmin, xmax, * FROM product;
-- ctid  | xmin | xmax | product_id | name                 | price | stock
-- (0,1) | 3969 |    0 |          1 | Wireless Mouse       | 25.99 |   100
-- (0,2) | 3969 |    0 |          2 | USB-C Hub            | 45.00 |    50

When a row is updated under MVCC, a new version is created at a new ctid. The old version stays until Vacuum cleans it up.

4. Isolation Anomalies — The 6 Problems of Concurrent Transactions

When multiple transactions run concurrently without proper isolation, several types of anomalies (bugs) can occur. The SQL standard defines 4 classic anomalies; there are also 2 additional serialization anomalies that PostgreSQL handles:

4.1 Dirty Write

T1 writes a value. Before T1 commits, T2 overwrites that value and commits. Then T1 rolls back. Now T2's committed value has been rolled back too — but T2 doesn't know!

Example:
T1: BEGIN; UPDATE product SET price = 50 WHERE product_id = 1; (not committed)
T2: BEGIN; UPDATE product SET price = 60 WHERE product_id = 1; COMMIT;
T1: ROLLBACK; — This rolls back both writes! Price goes back to original, losing T2's commit.

Prevented by: All isolation levels (even Read Uncommitted in standard SQL) prevent dirty writes. PostgreSQL never allows dirty writes.

4.2 Dirty Read

T1 reads data written by T2, but T2 has not yet committed. T2 may later rollback, meaning T1 read data that "never happened".

Example:
T2: BEGIN; UPDATE product SET stock = 0 WHERE product_id = 3;
T1: SELECT stock FROM product WHERE product_id = 3; → reads 0 (uncommitted!)
T2: ROLLBACK; — The stock was never really 0, but T1 already acted on that value.

Prevented by: Read Committed and above. PostgreSQL's minimum isolation level is Read Committed so dirty reads never happen in PostgreSQL.

4.3 Non-Repeatable Read (NRR)

T1 reads a row. T2 updates and commits that row. T1 reads the same row again within the same transaction and gets a different value.

Example (from slides — headphone inventory):
T1: SELECT stock FROM product WHERE product_id = 5; → returns 10
T2: UPDATE product SET stock = 8 WHERE product_id = 5; COMMIT;
T1: SELECT stock FROM product WHERE product_id = 5; → returns 8 (different!)
T1 sees two different values for the same row within one transaction.

Prevented by: Repeatable Read and above.

4.4 Phantom Read (PR)

T1 runs a query with a WHERE condition and gets a set of rows. T2 inserts new rows (or deletes rows) that match T1's WHERE condition. T1 runs the same query again and gets a different set of rows — rows that "appeared" or "disappeared".

Example:
T1: SELECT * FROM product WHERE price < 100; → 5 rows
T2: INSERT INTO product(name, price, stock) VALUES ('Cable', 9.99, 200); COMMIT;
T1: SELECT * FROM product WHERE price < 100; → 6 rows (a phantom appeared!)

Prevented by: Serializable (in SQL standard). In PostgreSQL, also prevented by Repeatable Read (due to MVCC snapshot)!

4.5 Lost Update

Two transactions both read the same value, compute an update based on it, and write back. One transaction's update overwrites (loses) the other's.

Example (SSD stock):
T1: reads stock = 10, computes 10 - 1 = 9
T2: reads stock = 10, computes 10 - 1 = 9
T1: UPDATE product SET stock = 9; COMMIT;
T2: UPDATE product SET stock = 9; COMMIT; ← This is wrong! Two items sold but stock only decreased by 1.

Prevented by: Repeatable Read in PostgreSQL (NOT in SQL standard!). In standard SQL RR, Lost Update is possible; PostgreSQL's MVCC RR prevents it by aborting the second writer.

4.6 Write Skew

Two transactions both read a set of rows to check a condition, and each writes to different rows based on what they read — but together their writes violate a global invariant that neither write alone would violate.

Example (from slides — promotion invariant):
Business rule: At most ONE product can have promotion = true.

T1: SELECT * FROM product WHERE promotion = true; → 0 rows (no promotions active)
T2: SELECT * FROM product WHERE promotion = true; → 0 rows (no promotions active)
T1: UPDATE product SET promotion = true WHERE product_id = 4; COMMIT; ✓ (1 promotion)
T2: UPDATE product SET promotion = true WHERE product_id = 6; COMMIT; ✓ (1 promotion)

Result: 2 products now have promotion = true. Invariant violated!
Neither T1 nor T2 individually violated anything — they each promoted exactly one product after checking none were promoted. But concurrently they both read "0 promotions" and acted on that.

Prevented by: Only Serializable isolation level.

5. Isolation Levels

5.1 Setting Isolation Levels in PostgreSQL

-- Set for current transaction (must be right after BEGIN):
BEGIN;
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;   -- default
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Or in shorthand:
BEGIN ISOLATION LEVEL REPEATABLE READ;
BEGIN ISOLATION LEVEL SERIALIZABLE;

5.2 Read Committed (PostgreSQL Default)

  • Each statement within the transaction gets a fresh snapshot — it sees all data committed before that statement started
  • If T2 commits between two SELECTs in T1, T1's second SELECT sees T2's changes
  • Prevents: Dirty Writes, Dirty Reads
  • Does NOT prevent: NRR, Phantom Read, Lost Update, Write Skew
RC behavior example:
T1 (RC): BEGIN; SELECT stock FROM product WHERE product_id=3; → 40
T2: UPDATE product SET stock = 38 WHERE product_id=3; COMMIT;
T1 (RC): SELECT stock FROM product WHERE product_id=3; → 38 (T2's commit is visible!)
T1: COMMIT;

T1 saw two different values — this is a Non-Repeatable Read. Allowed at RC.

5.3 Repeatable Read (PostgreSQL)

  • The entire transaction uses a snapshot taken at the moment the first statement executes
  • T1 always sees the same data regardless of what other transactions commit during T1's execution
  • If a write-write conflict occurs (T2 commits an update to a row that T1 also wants to update), T1 gets an error: ERROR: could not serialize access due to concurrent update and must retry
  • Prevents: Dirty Write, Dirty Read, NRR, Phantom Read, Lost Update (PostgreSQL-specific)
  • Does NOT prevent: Write Skew
RR behavior example:
T1: BEGIN ISOLATION LEVEL REPEATABLE READ; SELECT stock FROM product WHERE product_id=3; → 40
T2: UPDATE product SET stock = 38 WHERE product_id=3; COMMIT;
T1: SELECT stock FROM product WHERE product_id=3; → still 40 (snapshot!)
T1: COMMIT;

T1 always sees 40 — the snapshot was fixed at T1's start. Repeatable Read is maintained.

5.4 Serializable (PostgreSQL SSI)

  • Highest isolation level — guarantees that concurrent transactions appear to have executed serially
  • Uses MVCC (like RR) + additional predicate locking to detect rw-dependencies
  • If PostgreSQL detects that a set of concurrent transactions could not have produced the same result in any serial order, it aborts one with: ERROR: could not serialize access due to read/write dependencies among transactions
  • Prevents all anomalies including Write Skew
  • More overhead than RR — predicate locks must be tracked

6. Concurrency Control: 2PL vs MVCC Overview

There are two fundamentally different approaches to concurrency control:

Property 2PL (Two-Phase Locking) MVCC (Multiversion CC)
Approach Pessimistic — assume conflicts will happen, block preemptively Optimistic — allow reads always, detect conflicts on write
Read blocks Write? Yes — S-lock blocks X-lock requests No — reads never block writes
Write blocks Read? Yes — X-lock blocks S-lock requests No — writes never block reads
Guarantees Serializability? Yes (strict 2PL) No (only with Serializable/SSI)
Age Introduced ~1976 (50 years ago) Introduced ~1996 (30 years ago)
Used by MS SQL Server (default), MySQL/MariaDB (partly) PostgreSQL, Oracle, MySQL InnoDB, SAP HANA
Key insight One version of each row; lock it to access it Multiple versions of each row; choose correct version by timestamp
Why MVCC wins in practice: The key advantage of MVCC is that reads never block writes and writes never block reads. In a typical web application, reads are far more frequent than writes. With 2PL, every read blocks a potential write and vice versa, creating serious performance bottlenecks at scale. MVCC avoids this by keeping multiple versions.

7. Lock-Based Concurrency Control (2PL) — Deep Dive

7.1 Lock Modes and Compatibility

There are two basic lock modes in lock-based CC:

  • S (Shared / Read Lock): Acquired when a transaction wants to READ an object. Multiple transactions can hold S-locks simultaneously — reads don't conflict with reads.
  • X (Exclusive / Write Lock): Acquired when a transaction wants to WRITE an object. Only one transaction can hold an X-lock at a time, and it blocks all other S and X lock requests.

Lock Compatibility Matrix

Requester → S (wants to read) X (wants to write)
Holder: S (reading) ✓ Compatible ✗ Block
Holder: X (writing) ✗ Block ✗ Block

Summary: SS = compatible (both can read). SX, XS, XX = block (wait).

7.2 Lock Granularity

Locks can be applied at different granularities:

  • Table-level locks — easy to manage, low overhead, but very coarse (entire table locked)
  • Page-level locks — medium granularity
  • Row-level locks — fine-grained, allows high concurrency (different rows can be locked by different transactions)

PostgreSQL uses row-level locks for DML (INSERT/UPDATE/DELETE) and table-level locks for DDL (ALTER TABLE, DROP TABLE).

7.3 Why Simple Locking is Not Enough — The Airline Example

Consider an airline serving two routes: Kutaisi–Munich and Kutaisi–Berlin, both costing €100. Business invariant: both routes always cost the same. Two applications run concurrently:

  • App1 (A1): Increase price of both routes by 10%
  • App2 (A2): Add €10 CO₂ tax to both routes

Serial execution results: Either 120€ (A2 then A1: 100+10=110, 110×1.1=121 — wait, or A1 then A2: 100×1.1=110, 110+10=120). Both routes stay equal.

With naive locking (locks released immediately after each operation):

Op#T1 (App1: ×1.1)T2 (App2: +10)
1BEGIN; s-lock(a); read(a)=100
2x-lock(a); write(a)=110; unlock(a)
3BEGIN; s-lock(a); read(a)=110; x-lock(a); write(a)=120; unlock(a)
4s-lock(b); read(b)=100; x-lock(b); write(b)=110; unlock(b); COMMIT
5s-lock(b); read(b)=110; x-lock(b); write(b)=121; unlock(b); COMMIT

Result: a=120, b=121 — the two routes have different prices! The invariant is violated. This result is not achievable by any serial execution.

Problem: Locks were released too early. T1 unlocked (a) before it finished, allowing T2 to interleave in a non-serializable way. We need a protocol that guarantees serializability.

7.4 Two-Phase Locking (2PL)

2PL Rule: Split each transaction into two phases:
  1. Growth Phase (Lock Phase): Acquire all needed locks. During this phase, NO lock may be released.
  2. Release Phase (Shrink Phase): Release locks. Once you release the first lock (starting the shrink phase), you can NEVER acquire a new lock.
"No LOCK after an UNLOCK."

The "lock point" is the peak — the moment when a transaction holds all its locks. Any conflicting transaction must either wait (if it hasn't reached its lock point yet) or is already releasing locks (and therefore no longer acquiring new ones).

Key theorem: If every transaction in a schedule follows 2PL, the entire schedule is guaranteed to be serializable.

7.5 Strict 2PL — How Commercial DBMS Actually Do It

Basic 2PL allows releasing locks during the shrink phase, but this can cause cascading rollbacks: if T1 releases a lock and T2 reads T1's data, then T1 aborts — T2 must also abort because it read uncommitted data.

Commercial databases avoid cascading rollbacks by using Strict 2PL:

Strict 2PL: ALL locks are held until the end of the transaction (COMMIT or ROLLBACK). Locks are only released at commit/rollback, not during the transaction.

The growth phase goes all the way to commit. There is no shrink phase until the very end.
PostgreSQL note: PostgreSQL does NOT implement 2PL. It uses MVCC. However, PostgreSQL always holds its (write) locks until transaction end — like strict 2PL would. This is because in MVCC, PostgreSQL uses locks only for write-write conflicts (not for reads), and keeps those write locks until commit.

7.6 2PL Solves the Airline Problem

With strict 2PL applied to the airline example:

Op#T1 (App1: ×1.1)T2 (App2: +10)
1BEGIN; s-lock(a)
2read(a)=100; x-lock(a)
3write(a)=110; s-lock(b)BEGIN → tries s-lock(a) → BLOCKED (T1 holds x-lock!)
4read(b)=100; x-lock(b); write(b)=110⏳ waiting...
5COMMIT → releases all locks
6s-lock(a) granted; read(a)=110; x-lock(a); write(a)=120
7s-lock(b); read(b)=110; x-lock(b); write(b)=120; COMMIT

Result: a=120, b=120 — both routes cost the same. Serializable! (Equivalent to T1 first, then T2.)

7.7 2PL Schedule Analysis — Can These Run Under 2PL?

Exam Exercise Type: "Would this schedule run under 2PL?"
Schedule 1:
T1T2
Begin; Read(A)
Begin; Read(A)
Commit
Read(A); Commit
T1 holds S-lock(A) at first Read. T2 requests S-lock(A) — SS is compatible! Both can hold S-locks simultaneously. T2 commits, releasing its S-lock. T1 reads A again (still holds S-lock). Commits. YES, this runs under 2PL.
Schedule 2:
T1T2
Begin; Read(A)
Begin; Write(A); Commit

T1 holds S-lock(A). T2 requests X-lock(A) — SX conflicts! T2 is BLOCKED until T1 releases its S-lock. But under strict 2PL, T1 holds its S-lock until commit. So T2 must wait for T1 to commit. Under 2PL, T2 blocks until T1 commits. The schedule as written (T2 commits before T1's second read) cannot happen under 2PL — it would be serialized as T1 then T2.

8. MVCC — Multi-Version Concurrency Control Deep Dive

8.1 The Core Idea of MVCC

MVCC fundamental rule: When a transaction writes an object, a new version of that object is created. Old versions are kept. When a transaction reads, it reads the appropriate version for its snapshot — not necessarily the latest version.

This means:

  • Readers never block writers — a read transaction reads an old version, the writer creates a new version; no conflict
  • Writers never block readers — same reason
  • Only write-write conflicts require coordination (one writer must wait for the other)

Comparison: 2PL vs MVCC Lock Table

Object A — who holds what2PL: T2 wants to Read2PL: T2 wants to Write
T1 is Reading (holds S-lock)✓✗ Block
T1 is Writing (holds X-lock)✗ Block✗ Block
Object A — who holds whatMVCC: T2 wants to ReadMVCC: T2 wants to Write
T1 is Reading✓✓ (reads old version)
T1 is Writing✓ (reads old version)✗ Block (write-write)

8.2 MVCC Version Chain — The Basics

In PostgreSQL, every row has two hidden system columns:

  • xmin (created_by): the transaction ID (txid) of the transaction that created this version
  • xmax (deleted_by): the transaction ID that deleted/superseded this version (0 = still valid)

Operations and Version Effects

Concrete Example

-- Initial state (inserted by transaction 3939):
Version A0: value=x, xmin=3939, xmax=0  (valid)

-- Transaction 4066 updates it:
Version A0: value=x, xmin=3939, xmax=4066  (superseded)
Version A1: value=y, xmin=4066, xmax=0    (current)

To check what happened:

SELECT ctid, xmin, xmax, * FROM customer;
-- If xmax=0: row is currently valid
-- If xmax=4066: row was deleted/updated by transaction 4066
-- Check status: SELECT pg_xact_status('4066'::xid8);
-- → 'committed' means A0 is dead, A1 is live
-- → 'aborted' means A0 is still live, A1 is gone

8.3 MVCC Visibility Rules

Read Committed — Per-Statement Snapshot

A transaction in RC reads the latest committed version as of when each statement started.

Visibility rules for a READ statement:
  1. Writes by transactions that were active (not committed) when this statement started → ignored. If they later commit, they become visible to the next statement.
  2. Writes by aborted transactions → ignored
  3. Writes by transactions that committed AFTER this transaction started (but before this statement started) → visible
  4. All other committed writes → visible

Repeatable Read — Per-Transaction Snapshot

A transaction in RR takes one snapshot at the moment its FIRST statement runs, and uses that snapshot for all subsequent statements.

Visibility rules for RR:
  1. Writes by transactions that were active when this transaction started → ignored, even if they later commit
  2. Writes by aborted transactions → ignored
  3. Writes by transactions that started AFTER this transaction → ignored, regardless of commit status
  4. All other committed writes → visible

8.4 MVCC with the Kiushop Product Table — Step by Step

In the kiushop2026 database, product_id=3 is the Mechanical Keyboard with stock=40.

-- Session 1 (T1): Start and read
BEGIN;
SELECT txid_current_if_assigned();  -- Not yet assigned (no write done)
SELECT ctid, xmin, xmax, * FROM product WHERE product_id = 3;
-- ctid  | xmin | xmax | product_id | name                | price | stock
-- (0,3) | 3800 |    0 |          3 | Mechanical Keyboard | 89.99 |    40

-- Session 2 (T2): Update stock
BEGIN;
SELECT ctid, xmin, xmax, * FROM product WHERE product_id = 3; -- same: stock=40
UPDATE product SET stock = stock - 2 WHERE product_id = 3;
SELECT txid_current_if_assigned();  -- → 4051 (T2 got a txid)
SELECT ctid, xmin, xmax, * FROM product WHERE product_id = 3;
-- ctid  | xmin | xmax | ...
-- (0,8) | 4051 |    0 |   3 | Mechanical Keyboard | 89.99 | 38  ← NEW version

-- Session 1 (T1): Read again BEFORE T2 commits
SELECT ctid, xmin, xmax, * FROM product WHERE product_id = 3;
-- UNDER RC: still sees (0,3) stock=40 because T2 not committed yet
-- (T2's xmin=4051 is active → ignored)

-- Session 2 (T2): Commit
COMMIT;

-- Session 1 (T1): Read again AFTER T2 commits
SELECT ctid, xmin, xmax, * FROM product WHERE product_id = 3;
-- UNDER RC: NOW sees (0,8) stock=38 → T2's commit is visible to new statement
-- UNDER RR: STILL sees (0,3) stock=40 → snapshot was fixed at T1 start, T2's txid > T1's snapshot
T1: COMMIT;
Key difference RC vs RR in MVCC:
RC: "fresh snapshot per statement" → sees T2's commit after T2 commits
RR: "fixed snapshot per transaction" → never sees T2's changes (even after T2 commits), because T2 started after T1

8.5 MVCC Write-Write Conflicts

Even though MVCC avoids read-write conflicts, two writers on the same row still conflict. PostgreSQL uses exclusive row-level locks for writes.

Under Read Committed (RC):

T1 writes row A (sets xmax=T1_txid, creates A1 with xmin=T1_txid). T2 also wants to write row A:

  1. T2 sees the X-lock on A (T1's write is in progress) and WAITS
  2. T1 commits → A0 is officially deleted, A1 is the new version
  3. T2 wakes up, gets a fresh snapshot (RC!), now sees A1
  4. T2 can write A1 to create A2 — but this may cause a Lost Update if T2's update was based on the original A0 value

Under Repeatable Read (RR):

  1. T2 waits for T1's write (same as RC)
  2. T1 commits
  3. T2 wakes up. In RR, T2 sees A0 (its snapshot is fixed — T1's update is invisible to T2's snapshot).
  4. But T2 sees that A0's xmax is T1's committed transaction ID. A0 is dead AND there's a newer version T2 can't see.
  5. PostgreSQL ABORTS T2 with: ERROR: could not serialize access due to concurrent update
  6. T2 must retry its transaction from scratch.
Why this prevents Lost Updates under RR: In the headphone example, T1 sets stock=9 (was 10, sold 1). T2 also read stock=10 and computed 9. Under RR, T2 is aborted when T1 commits, preventing T2 from blindly writing 9 on top of T1's committed change. T2 retries, reads the new stock=9, computes 8, and writes 8 correctly.

9. Deadlocks — Detection, Prevention & PostgreSQL Handling

9.1 What Is a Deadlock?

A deadlock occurs when two or more transactions are each waiting for a lock held by the other — creating a circular dependency. None of them can proceed because each is waiting for a resource the other holds.

Classic 2-transaction deadlock (from slides — Airline example with strict 2PL):

Op#T1 (×1.1 on a and b)T2 (+10 on a and b)
1BEGIN
2BEGIN
3x-Lock(a) ✓; write(a)=a×1.1
4x-Lock(b) ✓; write(b)=b+10
5tries x-Lock(b) → BLOCKED (T2 holds b)
6⏳ waiting...tries x-Lock(a) → BLOCKED (T1 holds a)
7☠️ DEADLOCK — both waiting forever
Important from slides: Deadlocks happen not only under 2PL but also under MVCC! PostgreSQL uses MVCC for reads but uses row-level locks for writes. Two transactions can deadlock on write locks.

9.2 Waits-For Graph (WFG)

The standard method to detect deadlocks is the Waits-For Graph:

  • Each active transaction is a node
  • An edge from T1 to T2 means "T1 is waiting for a lock held by T2"
  • A cycle in the graph means a deadlock exists

Example from slides — a 3-transaction deadlock in PostgreSQL:

ERROR: deadlock detected
DETAIL: Process 1456 waits for ShareLock on transaction 1700; blocked by process 8264.
        Process 8264 waits for ShareLock on transaction 1701; blocked by process 10628.
        Process 10628 waits for ShareLock on transaction 1702; blocked by process 1456.

Waits-For Graph cycle: 1456 → 1700 → (held by 8264) → 8264 → 1701 → (held by 10628) → 10628 → 1702 → (held by 1456) → back to 1456. Cycle! Deadlock.

In PostgreSQL, a transaction ID gets its own lock (a ShareLock on the transaction ID itself). When T2 must wait for T1 to finish (because T1 holds a write lock T2 needs), T2 acquires a ShareLock on T1's transaction ID. T1 releases this lock at commit/rollback, signaling T2 to proceed.

9.3 How PostgreSQL Detects & Handles Deadlocks

PostgreSQL uses an optimistic approach — it does NOT try to prevent deadlocks (prevention is too costly). Instead:

  1. When a transaction waits for a lock, PostgreSQL sets a Deadlock-Timeout (default: 1 second)
  2. After 1 second without progress, PostgreSQL runs the Waits-For-Graph Algorithm to check for a cycle
  3. If a cycle is found, PostgreSQL picks one transaction as the victim and rolls it back
  4. The victim gets an error: ERROR: deadlock detected
  5. The other transactions can then proceed

Why wait 1 second before checking? The WFG algorithm is computationally expensive. PostgreSQL bets that most waits are just normal blocking (not deadlocks) and will resolve quickly. Only if a wait exceeds 1 second is it worth running the expensive check.

9.4 Preventing Deadlocks

Method 1: Preclaiming (Conservative 2PL)

All locks that a transaction might ever need are acquired at the very start (before any operation).

  • Cannot deadlock — transaction has all its locks before anyone else starts
  • Problem: Transaction must know ALL objects it will touch at start time — often impossible. Also severely limits concurrency — objects are locked far longer than needed.
  • Also called Conservative 2PL

Method 2: Timestamp-Based (Wait/Die and Wound/Wait)

Each transaction gets a timestamp when started. Older transactions have lower timestamps. In case of conflict:

  • Wait/Die: If T1 (older) wants a lock held by T2 (younger): T1 WAITS. If T2 (younger) wants a lock held by T1 (older): T2 DIES (rollbacks and retries later). Older transactions always get priority.
  • Wound/Wait: Opposite — if an older transaction wants a lock, it WOUNDS (preempts) the younger. The younger dies.

Commercial DBMS typically use WFG detection rather than timestamp-based prevention because prevention requires knowing timestamps upfront and limits concurrency.

9.5 Dealing with Deadlocks from the Application Side

Since deadlocks can't be fully prevented in practice, applications must be prepared:

  1. Detect deadlock error in application code (SQLSTATE 40P01 in PostgreSQL)
  2. Retry the transaction from scratch
  3. Keep transactions short — less time holding locks = less chance of deadlock
  4. Access objects in the same order — if T1 always locks table A before B, and T2 does too, they can never deadlock on A and B
  5. Avoid long interactive transactions — don't leave a BEGIN open for minutes waiting for user input
  6. Limit use of SELECT ... FOR UPDATE — it acquires write locks on rows prematurely

  • All physical versions of a logical tuple are stored in the same table (heap file)
  • New versions are appended to the table (or stored in freed-up space after vacuum)
  • Versions of different rows are interleaved on the same pages

  1. Do NOT put indexes on heavily-updated columns (like score, stock, price) — this forces expensive COLD updates that require writing to all indexes
  2. Use fill factor < 100% to leave space on pages for HOT update versions:

10.3 Vacuum — Garbage Collection

Over time, old versions of rows accumulate in the heap. Vacuum is PostgreSQL's garbage collector that removes dead tuples:

VACUUM product;          -- reclaim space from dead tuples
VACUUM ANALYZE product;  -- also updates table statistics

-- Check dead tuples before vacuum:
CREATE EXTENSION IF NOT EXISTS pgstattuple;
SELECT * FROM pgstattuple('product'::regclass);

-- Check HOT update statistics:
SELECT relname, n_tup_upd, n_tup_del, n_tup_hot_upd, n_tup_newpage_upd
FROM pg_stat_user_tables WHERE relname = 'product';

What Vacuum Does

  1. Identifies dead tuples: Rows where xmax is a committed transaction and no active transaction can still see the old version (xmax < oldest active snapshot)
  2. Reclaims heap space: Marks dead line-pointer slots as free (available for new inserts/updates)
  3. Updates line pointers: After vacuum, line pointers that were in a HOT chain are converted to "re-direct" pointers, so indexes can still find the current version
  4. Cleans up Btree indexes: Removes dead index entries pointing to dead tuples

10.4 MVCC in Other DBMS

MySQL/MariaDB/Oracle: Delta (Undo Log) Storage

  • The main table always contains the current version only
  • On every update, the old version is written to the Undo Log
  • The current row has a pointer to its most recent undo log entry
  • Undo log entries chain from newest to oldest (for transactions that need very old versions)
  • Secondary indexes always point directly to the current version in the main table
  • Advantage vs PostgreSQL Append-Only: No index entries need to be changed on update (indexes always point to the same row in the main table)
  • Disadvantage: Reading old versions requires following the undo chain (potentially slow if many old versions)

SAP HANA: Time-Travel Storage

  • On every update, the full old version is inserted into a separate Time-Travel table (history table)
  • The main table always holds the current version
  • Versions are linked by PK + validity interval metadata (valid_from, valid_to)
  • No pointers needed — you find old versions by searching the time-travel table by PK + time

11. SELECT ... FOR UPDATE — Explicit Pessimistic Locking

11.1 What is SELECT ... FOR UPDATE?

By default PostgreSQL uses MVCC where reads don't block writes. But sometimes you need to explicitly lock rows before modifying them. SELECT ... FOR UPDATE is the tool for this:

SELECT ... FOR UPDATE acquires an exclusive write lock on all returned rows immediately at SELECT time — before any UPDATE is executed. Other transactions that want to write those rows must wait. 
BEGIN;
SELECT * FROM product WHERE product_id > 6 FOR UPDATE;
-- Locks all rows where product_id > 6 immediately
UPDATE product SET stock = 0 WHERE product_id > 6;
COMMIT;
-- Locks are released at COMMIT

11.2 Why Use SELECT ... FOR UPDATE?

Typical use cases:

  • Protecting a Read-Modify-Write cycle: Read a value, compute a new value based on it, write the new value. You want to ensure the value hasn't changed between your read and write.
  • Avoiding Lost Updates (under RC where MVCC alone doesn't prevent them): Explicitly lock the row you read so no one else can modify it before you write.
  • Preventing Write Skew (partially): Lock all rows that your condition depends on, so another transaction that reads the same rows must wait.
  • Avoiding retries: Under RR, write-write conflicts cause transaction aborts + retries. FOR UPDATE at RC level can block instead of abort.

11.3 Costs of SELECT ... FOR UPDATE

  • Higher deadlock risk — holding more locks for longer increases deadlock probability
  • Lower concurrency — other transactions must wait for the locked rows
  • Extra disk writes — PostgreSQL must record that a transaction holds a lock on these rows; this involves extra WAL writes
Rule of thumb: SELECT ... FOR UPDATE adds pessimistic coordination but costs performance and deadlock risk. Think twice before using it. Consider whether a higher isolation level (RR or Serializable) would solve the problem more cleanly.

11.4 Simulated Example

Suppose we want to safely reduce stock when a product is ordered (protect against overselling under RC):

-- UNSAFE at RC (Lost Update possible):
BEGIN;
SELECT stock FROM product WHERE product_id = 3; -- reads stock=40
-- Another transaction runs here and sets stock=39
UPDATE product SET stock = 39 WHERE product_id = 3; -- WRONG: stock should be 38
COMMIT;

-- SAFE with FOR UPDATE:
BEGIN;
SELECT stock FROM product WHERE product_id = 3 FOR UPDATE;
-- Other transactions wanting to update product_id=3 now BLOCK here
-- stock=40, safely locked
UPDATE product SET stock = stock - 1 WHERE product_id = 3;
COMMIT; -- releases the lock, stock=39 correctly

12. Write Skew Anomaly — Deep Dive

12.1 Definition

Write Skew occurs when:

  1. Two (or more) transactions each read an overlapping set of data
  2. Each makes a write decision based on what they read
  3. They write to different rows (so no direct write-write conflict)
  4. Together, their writes violate a global invariant

The tricky part: neither transaction individually violates anything. The problem is their combined effect based on each reading a state that becomes stale due to the other's write.

12.2 Write Skew Example (Promotion Invariant)

Business invariant: At most ONE product can have promotion = TRUE at any time.

-- Setup:
ALTER TABLE product ADD COLUMN promotion BOOLEAN DEFAULT false;
-- All products now have promotion = false
T1 (in terminal 1): 
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT * FROM product WHERE promotion = true;
-- (0 rows) — no promotions active

UPDATE product SET promotion = true WHERE product_id = 4;

SELECT * FROM product WHERE promotion = true;
-- (1 row) — I promoted product 4

COMMIT;
T2 (in terminal 2, running concurrently): 
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT * FROM product WHERE promotion = true;
-- (0 rows) — no promotions active (same snapshot as T1's start)

UPDATE product SET promotion = true WHERE product_id = 6;

SELECT * FROM product WHERE promotion = true;
-- (1 row) — I promoted product 6

COMMIT;

Result: Both transactions commit successfully. Now promotion = true for BOTH product 4 and product 6. The invariant is violated!

Why does this happen? T1 and T2 each read "0 promotions" from their RR snapshot. They each write to a different row (product 4 vs product 6) — no write-write conflict. PostgreSQL doesn't detect any problem. Both commit. But the combined result violates the business rule.

12.3 SELECT ... FOR UPDATE Does NOT Fully Fix Write Skew

Using SELECT ... FOR UPDATE works for the "update existing rows" case:

-- T1:
BEGIN;
SELECT * FROM product WHERE promotion = true FOR UPDATE;
-- Locks all rows where promotion=true (currently 0 rows — nothing to lock!)
UPDATE product SET promotion = true WHERE product_id = 4;
COMMIT;

-- T2 runs at the same time:
BEGIN;
SELECT * FROM product WHERE promotion = true FOR UPDATE;
-- Also locks 0 rows (no promoted products yet)
UPDATE product SET promotion = true WHERE product_id = 6;
COMMIT;

Problem: When there are 0 promotions, FOR UPDATE locks 0 rows — there's nothing to lock! T1 and T2 both lock an empty set and proceed to promote different products. Write skew still occurs.

CRITICAL: If T2 inserts a NEW product that is immediately promoted (not updating an existing row), SELECT ... FOR UPDATE CANNOT prevent the write skew. The new row didn't exist when T1 locked — T1 has no lock on a row that doesn't exist yet. This is called a "Phantom Insert" write skew and can only be prevented by Serializable.

12.4 Serializable Prevents Write Skew

-- T1:
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT * FROM product WHERE promotion = true; -- (0 rows)
UPDATE product SET promotion = true WHERE product_id = 4;
SELECT * FROM product WHERE promotion = true; -- (1 row)
COMMIT; -- SUCCESS

-- T2 (concurrent):
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT * FROM product WHERE promotion = true; -- (0 rows)
UPDATE product SET promotion = true WHERE product_id = 6;
SELECT * FROM product WHERE promotion = true; -- (1 row)
COMMIT; -- FAILS with: ERROR: could not serialize access due to
         --              read/write dependencies among transactions

PostgreSQL's SSI detects the rw-dependency cycle: T2 read data (promotion=true rows) that T1 wrote to, AND T1 read data that T2 wrote to. This circular read-write dependency is what SSI aborts.

Write Skew solutions ranked:
  1. BEGIN ISOLATION LEVEL SERIALIZABLE — cleanest, prevents all write skew (even phantom insert cases), but requires retry logic and has performance overhead
  2. SELECT ... FOR UPDATE — works for "update existing rows" write skew, but NOT for phantom insert write skew. Lower overhead than Serializable.
  3. Application-level locking (pessimistic advisory locks) — possible but complex

13. PostgreSQL Serializable — SSI (Serializable Snapshot Isolation)

13.1 How Serializable Works in PostgreSQL

PostgreSQL's Serializable isolation works exactly like Repeatable Read, PLUS it additionally monitors for rw-dependencies (read-write dependencies) between transactions.

rw-dependency: "If T2's write would have changed the result of T1's read, had T2 run before T1."

Two transactions with a circular rw-dependency cannot have been executed serially — so PostgreSQL must abort one of them.

13.2 Predicate Locking

To track rw-dependencies without blocking, PostgreSQL uses predicate locks:

  • Predicate locks are NOT blocking locks — they don't prevent access
  • They track what data each Serializable transaction READ (the "predicate" — the WHERE conditions)
  • If another transaction writes data that falls within a predicate lock's range, PostgreSQL knows there's an rw-dependency
  • Predicate lock types: page locking, gap locking, next-key locking

13.3 Error Messages: RR vs Serializable

SituationError Message
RR: Two transactions try to UPDATE the same row ERROR: could not serialize access due to concurrent update
Serializable: rw-dependency cycle detected (Write Skew) ERROR: could not serialize access due to read/write dependencies among transactions

13.4 Summary — All Isolation Levels in PostgreSQL

Level Snapshot Write-Write Extra Anomalies prevented
READ COMMITTED Per statement Wait for writer, get fresh snapshot — Dirty Write, Dirty Read
REPEATABLE READ Per transaction (at first statement) Wait for writer, then ABORT if committed — + NRR, Phantom Read, Lost Update
SERIALIZABLE Per transaction (at first statement) Wait for writer, then ABORT if committed Predicate locks, rw-dep tracking Everything including Write Skew

Waits-For Graph Cycle: 
T1 → (waiting for row locked by) → T2 → (waiting for row locked by) → T3 → (waiting for row locked by) → T1

COMMIT; -- SUCCESS

-- T2 (concurrent):
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT * FROM product WHERE promotion = true; -- 0 rows
INSERT INTO product(product_id, name, price, stock, promotion)
  VALUES (20, 'Projector', 2000, 20, true);
SELECT * FROM product WHERE promotion = true; -- 1 row
COMMIT; -- FAILS:
-- ERROR: could not serialize access due to read/write dependencies among transactions
-- HINT: The transaction might succeed if retried.

Why Serializable works: PostgreSQL's SSI tracks that T2 read the "no promotions" state that T1 wrote into. The rw-dependency cycle (T1 wrote what T2 read, T2 inserted a row that would change what T1 read) is detected, and T2 is aborted.

Exercise 5: ScyllaDB Consistency Behavior

Task
Demonstrate ScyllaDB consistency levels by starting 2 of 3 nodes, writing data at consistency ONE, then switching nodes, and observing that the write is NOT visible on the new node.
Conceptual explanation (paper exam format):

ScyllaDB is a peer-to-peer distributed database that uses consistent hashing to assign partitions to nodes, and a replication factor (RF) to duplicate each partition across RF nodes.

Setup: 3 nodes (kiu-scylla1, kiu-scylla2, kiu-scylla3), RF=3 (each partition on all 3 nodes). Start nodes 1 and 2 only (node 3 is down). Step-by-step answers:
  1. Default consistency = ONE — only 1 node needs to acknowledge a write/read
  2. Write to a partition stored on kiu-scylla2 at consistency ONE: succeeds (kiu-scylla2 is up)
  3. Stop kiu-scylla2, start kiu-scylla3
  4. Does the write show on kiu-scylla3? NO. The write was only sent to kiu-scylla2 at consistency ONE. kiu-scylla3 was down at write time and never received the update. The data is NOT replicated to kiu-scylla3 until node healing (hinted handoff, repair) occurs.
  5. Does the write show on kiu-scylla1? YES (if kiu-scylla1 is one of the replica nodes for that partition). kiu-scylla1 was up and received the ONE acknowledgment OR kiu-scylla2 forwarded to kiu-scylla1 as coordinator.
  6. Change consistency to QUORUM on kiu-scylla3: needs ⌊RF/2⌋+1 = 2 nodes to agree (with RF=3)
  7. Try to write to a partition on kiu-scylla2 (which is DOWN) at QUORUM: FAILS — only kiu-scylla1 and kiu-scylla3 are up. To write a partition that needs kiu-scylla2 and only has 1 other available node, QUORUM (2) cannot be reached → write error.
  8. Can you write at QUORUM with kiu-scylla2 down? YES — choose a partition stored on kiu-scylla1 AND kiu-scylla3 (not kiu-scylla2). With kiu-scylla1 and kiu-scylla3 both up, QUORUM=2 is achievable.
Key concepts:
  • Consistency ONE: Fast but weak — only 1 node must acknowledge. Writes may not reach all replicas immediately → stale reads possible on other nodes.
  • Consistency QUORUM: Majority of replicas must acknowledge. Stronger consistency but fails if majority is unavailable.
  • Hinted Handoff: When a replica is down, ScyllaDB stores a "hint" on the coordinator node and delivers it when the replica comes back online.
