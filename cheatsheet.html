MODULE 5 — PostgreSQL Transactions & Concurrency

A. ACID Properties

A — Atomicity: All-or-nothing. If any step fails → entire transaction is rolled back. No partial commits.
Implemented via: WAL (undo log) and rollback mechanism.

C — Consistency: DB moves from one valid state to another. All constraints (PK, FK, CHECK) hold before and after.

I — Isolation: Concurrent transactions appear to run serially. Implemented via 2PL or MVCC.
Levels: Read Committed (default) → Repeatable Read → Serializable.

D — Durability: Once COMMITTED, data survives crashes. Implemented via WAL flush to disk before COMMIT returns.
Which ACID property does PostgreSQL implement via WAL? → Durability. Atomicity uses the undo mechanism (old versions in heap). Isolation uses MVCC snapshots.

B. WAL & Buffer Pool

WAL (Write-Ahead Log) = sequential append-only log. Every change is written to WAL BEFORE the data page is written to disk.
- Stored in $PGDATA/pg_wal/
- Each record has an LSN (Log Sequence Number) — unique monotonically increasing 64-bit number
- On crash: PostgreSQL replays WAL from last checkpoint → no data loss
COMMIT flow: (1) Write COMMIT record to WAL → (2) fsync WAL to disk → (3) return "COMMIT" to client. Data pages may still be dirty in buffer pool — they'll be written later. WAL already guarantees durability.
Checkpoint: All dirty buffer pool pages flushed to disk. After crash, only replay WAL from last checkpoint (not from beginning). Happens every checkpoint_timeout (default 5 min) or when WAL fills (max_wal_size).

ctid = physical row address = (page_number, slot_number). Example: (0,1) = page 0, slot 1. Updates create new ctid.
WAL record types seen in pg_waldump: HOT_UPDATE = update stayed on same page, no index change. Plain UPDATE = indexed column changed OR no space → new ctid → index entries rewritten (INSERT_LEAF also appears).

RC = "fresh newspaper every statement". RR = "same newspaper all transaction". SERIALIZABLE = "newspaper + editor checks for contradictions between readers".

Strict 2PL (what commercial DBMS actually use): Hold ALL locks until COMMIT/ROLLBACK. No shrink phase until end. Prevents cascading rollbacks. PostgreSQL does NOT use 2PL — it uses MVCC. But PostgreSQL holds its write locks until commit (like Strict 2PL behavior for writes).
Airline 2PL example (hard exam case): T1: increase both routes by 10%. T2: add €10 tax to both routes. Naive locking (release early) → a=120, b=121 (different! invariant violated). With Strict 2PL: T2 blocks waiting for T1 to COMMIT (T1 holds X-lock on 'a'). T1 commits first → T2 proceeds → a=120, b=120. Serializable! ✓

F. MVCC Deep Dive — xmin / xmax

Every PostgreSQL row has hidden system columns:
xmin = transaction ID that created this version
xmax = transaction ID that deleted/updated this version (0 = still valid)
OperationEffect on versions
INSERTNew row: xmin=current_txid, xmax=0
DELETEOld row: xmax=current_txid (marked dead)
UPDATEOld row: xmax=current_txid; New row: xmin=current_txid, xmax=0
MVCC Visibility rule (Read Committed): Each statement sees data committed before that statement started. If T2's xmin > my statement's snapshot → invisible. If T2 committed after my statement started → invisible for this statement, visible for next statement.

MVCC Visibility rule (Repeatable Read): Snapshot fixed at first statement of transaction. Any transaction that started after my snapshot → permanently invisible, even after it commits.
Hard case — RR write-write conflict: T1 (RR) and T2 both want to UPDATE product stock. T2 commits first. T1 wakes up, sees xmax of the row = T2's committed txid. Row is dead, newer version exists. PostgreSQL ABORTS T1 with: ERROR: could not serialize access due to concurrent update. T1 must retry. This is how RR prevents Lost Updates!

G. Deadlocks

Deadlock = two or more transactions each waiting for a lock held by the other. Circular dependency → none can proceed.
Classic deadlock: T1 holds X-lock on row A, wants X-lock on B. T2 holds X-lock on B, wants X-lock on A. Both wait forever.
Waits-For Graph (WFG): Node per transaction. Edge T1→T2 means "T1 waits for lock held by T2". A cycle = deadlock.

PostgreSQL approach: Optimistic — does NOT prevent deadlocks. Instead: (1) Wait deadlock_timeout = 1 second. (2) Run WFG algorithm. (3) If cycle found → abort one victim transaction → others proceed. Error: ERROR: deadlock detected SQLSTATE 40P01.

Deadlock prevention strategies:
  • Preclaiming (Conservative 2PL) — acquire ALL locks at start. Never deadlocks but severely limits concurrency.
  • Timestamp (Wait/Die) — older transaction waits, younger dies. OR Wound/Wait: older wounds younger.
  • Application level — always acquire locks in same order, keep transactions short, retry on deadlock error.
Deadlocks happen even under MVCC! PostgreSQL uses MVCC for reads but exclusive row locks for writes → two writes can deadlock. The WFG is checked after 1 second wait.

H. HOT vs COLD Updates & Vacuum

Decision tree: Is the updated column indexed? YES → COLD. NO → Is there space on same page? YES → HOT. NO → COLD.

Practical tip: Don't put indexes on heavily-updated columns (stock, price, score) — every update becomes a COLD update and writes to ALL indexes. Use fillfactor=70 to leave space for HOT updates.
VACUUM: Removes dead tuples. (1) Identifies rows where xmax = committed txid and no active snapshot can still need the old version. (2) Reclaims heap space. (3) Converts HOT chain pointers to re-direct pointers. (4) Removes dead index entries.

I. SELECT … FOR UPDATE

SELECT ... FOR UPDATE acquires an exclusive write lock on returned rows immediately at SELECT time — before any UPDATE is executed. Other transactions wanting to write those rows must wait.
Use cases: protect read-modify-write cycles at RC, avoid lost updates, partially prevent write skew (for existing rows only).

FOR UPDATE does NOT fix phantom-insert write skew. If condition returns 0 rows (nothing to lock), FOR UPDATE locks nothing → write skew still possible. Only SERIALIZABLE can fix this.
Costs: higher deadlock risk, lower concurrency, extra WAL writes.

J. Write Skew & SSI

Write Skew pattern (memorize):
Rule: max 1 promotion. T1 reads promotions→0 rows. T2 reads promotions→0 rows. T1 sets product_4=promoted. T2 sets product_6=promoted. Both commit. Result: 2 promotions. Rule violated. Neither T1 nor T2 individually violated anything. Their combined reads+writes create the skew.
SSI (Serializable Snapshot Isolation) = PostgreSQL's Serializable: Uses MVCC snapshots PLUS predicate locks (non-blocking, just track what was read). If circular rw-dependency detected → abort one transaction.
Error: ERROR: could not serialize access due to read/write dependencies among transactions.

Write Skew fix ranking: (1) SERIALIZABLE — always works. (2) FOR UPDATE — works only when rows exist to lock. (3) Manual locking — complex.

MODULE 6.1 & 6.2 — P2P Distributed Databases

A. ACID in P2P & 3 Meanings of "Consistency"

ACID PropertyStatus in ScyllaDB/Cassandra (P2P)
AtomicityDoes NOT exist. No multi-statement / cross-partition transactions. Each CQL statement executes independently.
ConsistencyRedefined entirely — see below. NOT about DB constraints.
IsolationNo transactions → no isolation. BUT concurrent writes → conflict resolution needed.
DurabilityStrongest of all four — commit log + replication to multiple nodes.
3 DIFFERENT meanings of "Consistency" — never confuse them:
1. ACID Consistency = all DB constraints are satisfied (single-server RDBMS).
2. P2P Consistency = all replicas return the same value for the same request.
3. CAP Consistency = every node returns the most recent data — a global single view.
Exam: "Which ACID property is best supported by P2P?" → Durability. "Which doesn't exist?" → Atomicity.

B. CAP Theorem

CAP (Brewer 2000, proven 2002): A distributed replicated system can guarantee at most 2 of:
Consistency · Availability · Partition Tolerance
P is not optional — it simply happens. Networks fail. The REAL choice when partition occurs: keep C (refuse requests that can't guarantee latest data = reduced availability) OR keep A (answer from possibly stale data = weaker consistency).

ScyllaDB/Cassandra = A/P systems → Eventual Consistency.
Eventual Consistency: "Stop writing → wait some unspecified time → eventually all replicas return the same value." Weak guarantee but allows high availability + low latency.

Two causes of replica divergence (exam!):
  1. Write propagation delay (replication lag) — write reached some replicas but not all yet. Resolves by itself.
  2. Concurrent writes — two clients write same key via different coordinators simultaneously. No natural order → explicit conflict resolution needed.
Happened-Before: A happened before B if B knows about A (depends on A). If neither knows about the other → they are CONCURRENT. 3 cases: A before B, B before A, or concurrent. Concurrent writes have no natural order → system must impose one.

W + R > N rule: If write set and read set overlap in at least one replica, a read is guaranteed to see the latest write (no stale-only reads).
N−W = nodes that can be down while writes still work.

Critical edge case: Quorum write TIMES OUT → coordinator returns failure to client → but some replicas may have already applied the write. Applied writes are NOT rolled back. A "failed" write can still leave data behind!

D. Read Repair & Hinted Handoff

Read Repair: A read operation that finds different values across replicas: (1) Returns the value with the latest timestamp to client. (2) Triggers background write to update stale replicas. A read causes a write — drives toward eventual consistency.
Hinted Handoff: When a replica is DOWN during a write, the coordinator stores the write locally as a "hint" and delivers it to the down replica when it comes back online.

E. LWW — Last Write Wins

LWW: When concurrent writes conflict, the write with the highest timestamp wins. Loser is silently discarded even though it was acknowledged as successful.
LWW danger example: Client A and B both write to same key simultaneously. Both get "acknowledged: true". LWW picks one by timestamp — the other is silently deleted. Client whose write lost doesn't know. Data corruption.
Problems with LWW:
  • Uncoordinated clocks — each coordinator uses its own system clock. Clock drift/skew = node with faster clock always wins.
  • Silent data loss — all acknowledged writes except highest-timestamp are discarded.
  • Tie-break = undocumented behavior.
CQL commands: SELECT writetime(col) → cell's write timestamp. UPDATE ... USING TIMESTAMP N SET ... → forces a timestamp. If N < existing timestamp → write accepted but silently discarded (LWW comparison).
"All writes were acknowledged as successful — is data safe?" → NO with LWW. Acknowledged ≠ survived. Only the highest-timestamp write survives.

F. Version Vectors & Sibling Values

Version Vector (VV): A per-node counter list for each object. Tracks causal history. Lets the system determine if two versions are ordered (one is newer) or concurrent (both happened independently).
Rules:
  • VV_A ≤ VV_B in every slot → A happened-before B (B is newer). Sync to B.
  • VV_B ≤ VV_A in every slot → B happened-before A (A is newer). Sync to A.
  • Neither dominates → CONCURRENT writes → CONFLICT.
Concurrent VV example:
Node A has: {n2:2, n3:2, n4:1} (value="Alice Brown")
Node B has: {n2:1, n3:2, n4:2} (value="Alice Johnson")
Compare: n2 slot → A has 2, B has 1 (A bigger). n4 slot → A has 1, B has 2 (B bigger). Neither dominates → concurrent → SIBLING created!
CRITICAL exam answer: VVs can only DETECT concurrent conflicts. They cannot resolve them. (Multiple choice: answer C — "can only detect concurrent write conflicts")
Sibling Values (RIAK): With allow_mult=true, RIAK stores BOTH conflicting values as siblings. Both values AND both VVs kept. Application must resolve and pick one.
VV = "edit counter per node". Both counters go up on different nodes → concurrent → sibling. Only one went up → clear winner. LWW throws away causal history. VV preserves it.

G. CRDTs — Conflict-Free Replicated Data Types

CRDT = data structure where concurrent updates can ALWAYS be merged automatically without conflicts. Merge function is: Commutative (order doesn't matter) + Associative (grouping doesn't matter) + Idempotent (merging twice = merging once).

State-Based CRDT (6.3)

Sends full state. Merge = max per slot. Loose delivery (idempotent merge).
Example: G-Counter

Op-Based CRDT (6.4)

Sends operations (increment, add). Each op delivered exactly once per replica.
Example: OR-Set
G-Counter (Grow-Only Counter) — hard exam case:
Nodes n1, n2, n3. Each only increments its OWN slot.
n1: inc 2 → state (2,0,0)
n2: inc 3 → state (0,3,0)
Merge (max per slot): (2,3,0). Total = 2+3 = 5. ✓
Merge again: (2,3,0) merge (2,3,0) = (2,3,0). Idempotent ✓

WHY no lost update: each node owns its slot → concurrent increments land in different slots → both kept.
VV vs CRDT vector — DO NOT confuse:
VV = {key, value, VersionVector} — metadata about causality, detects conflicts.
CRDT vector = {key, CRDT_state} — the actual data values, no causality needed, auto-merges.
In CRDTs, time and causality are irrelevant — only the merged state matters.
Strong Eventual Consistency (SEC): Replicas that have delivered the same set of updates have identical state — no conflict resolution needed. State-based CRDTs satisfy SEC. Stronger than plain eventual consistency.

MODULE 6.3 & 6.4 — Consensus, 2PC, Raft, LWT, Paxos

A. Consensus

Consensus: All nodes agree on a value (or order of events or cluster decision) despite failures, using a majority quorum. One or more nodes propose, the algorithm decides on exactly one.
4 formal properties a correct consensus algorithm must have:
PropertyMeaning
AgreementNo two correct nodes decide differently
IntegrityNo node decides twice (decision is final)
ValidityDecided value was actually proposed (not invented)
TerminationEvery non-crashed node eventually decides (no infinite loop)
FLP Impossibility: No deterministic consensus algorithm guarantees both safety AND liveness in fully asynchronous network with even 1 crash. Systems get around this via partial synchrony + timeouts.

What consensus is used for in practice:
  • Leader election (MongoDB: Raft for choosing Primary)
  • Cluster/schema management (ScyllaDB: Raft for membership + schema)
  • Distributed transactions (2PC for cross-DB atomic commit)
  • Data uniqueness enforcement (ScyllaDB: Paxos for LWT)

B. Two-Phase Commit (2PC)

Wedding ceremony: priest asks "does anyone object?" (Phase 1 — PREPARE). If nobody objects → "you are married!" (Phase 2 — COMMIT). Even ONE objection → no wedding → ABORT.
2PC ≠ 2PL! 2PL = concurrency control within one database. 2PC = atomic commit protocol across multiple databases.
Phase 1 — PREPARE (Voting):
  1. Coordinator sends PREPARE to all participants
  2. Each participant: checks constraints, writes decision to WAL, replies YES or NO
  3. YES vote = binding promise — participant MUST commit when told, even after crash+recovery
  4. Coordinator collects all votes
Phase 2 — COMMIT or ABORT:
  1. ALL said YES → coordinator writes COMMIT to its log → sends COMMIT to all
  2. ANY said NO → coordinator sends ABORT to all
  3. Participants execute the decision, release locks, send ACK
All Failure Points:
When crashesWhoEffectResolution
Before PREPARE sentCoordinatorNothing done yetNew coordinator: abort cleanly
During PREPARE — no vote sentParticipantCoordinator times out → treats as NOAll abort
After voting YESCoordinatorDANGER: participants hold locks, cannot decide alone. BLOCKED.Must wait for coordinator to recover
During COMMIT — partial COMMITs sentCoordinatorSome committed, some didn'tParticipants query each other; coordinator retries on recovery
After voting YESParticipantCoordinator can still commit (has other votes)Down participant commits on recovery per coordinator instruction
The Blocking Problem (most important 2PC weakness): Participant voted YES → it is committed to that. Now it holds locks. If coordinator crashes before sending COMMIT/ABORT → participant is stuck indefinitely. 2PC is a blocking protocol.
True/False exam questions about 2PC:
A. "Coordinator is single point of failure" → TRUE
B. "Coordinator is single point for the final commit/abort decision" → TRUE
C. "Any node can block 2PC by crashing before voting YES" → TRUE (coordinator aborts)
D. "Any node can block 2PC by crashing AFTER voting YES" → FALSE — only the coordinator's crash blocks. If a participant crashes after YES, coordinator continues, commits, retries to the down node on recovery.
XA Standard: Standardized 2PC for heterogeneous DBs. Commands: XA START, XA END, XA PREPARE, XA COMMIT/XA ROLLBACK. PostgreSQL: PREPARE TRANSACTION 'id' then COMMIT PREPARED or ROLLBACK PREPARED.

Alternatives to 2PC:
  • 3PC: Adds "pre-commit" phase so participants can complete without coordinator. Requires synchronous messaging — essentially not used in practice (too complex).
  • Paxos-based 2PC: All nodes broadcast votes to each other. Each independently reaches same decision. No single coordinator → no single point of failure.

C. Raft Consensus Algorithm

A company needs decisions even when employees are sick. They elect a CEO (Leader). All decisions through CEO. CEO hit by bus? Vote new CEO and continue. That's Raft.
3 Roles:
RoleJobHow many
LeaderReceives ALL client requests. Replicates log. Sends heartbeats.Exactly 1 per term
FollowerPassive. Accepts entries from leader. Votes in elections.All others
CandidateTemporary during election. Campaigns for leadership.1+ during election
Leader Election:
  1. Time divided into Terms (1, 2, 3…). Each term has at most 1 leader.
  2. Follower has election timeout (random 150–300ms). No heartbeat in time → become Candidate.
  3. Candidate increments term, votes for self, sends RequestVote to all.
  4. Vote granted IF: (a) not voted this term yet, AND (b) candidate's log is at least as up-to-date.
  5. First to get majority (quorum) → becomes Leader.
  6. New leader immediately sends heartbeats to stop other elections.
Log Replication:
  1. Client request → Leader appends to log (uncommitted).
  2. Leader sends AppendEntries to all followers.
  3. Followers write entry, reply OK.
  4. Majority acked → Leader marks COMMITTED → applies to state machine → replies to client.
  5. Next heartbeat informs followers → they commit too.
Safety guarantees: At most 1 leader per term. Committed entries are never lost — any future leader of higher term will have them (Leader Completeness). Log Matching: same index+term → logs identical up to that point.
Hard case — Raft election during partition: 5 nodes. Leader A crashes. B and C have the committed log entry. D and E don't. B starts election: B votes self, C votes B (B's log is up-to-date), B gets majority → B becomes leader. D and E catch up on missing entries via AppendEntries.
Where Raft is used:
  • ScyllaDB — cluster membership + schema management (DDL operations)
  • MongoDB — Replica Set elections (choosing new Primary)
  • etcd (Kubernetes), CockroachDB, Consul, TiKV
Raft = Replicated log + election Timeout + majority vote. Leader wins election → owns all writes → majority confirms → commit. No leader → new election → continue.

D. LWT vs LWW — Lightweight Transactions

LWW silently discards data. LWT gives explicit rejection.
Why LWW fails for username registration:
Client A: INSERT username="flash99", user_id=111 (timestamp=1000)
Client B: INSERT username="flash99", user_id=222 (timestamp=1001)
LWW result: user_id=222 wins. user_id=111 silently deleted. Client A got "OK" but their account is gone. → CORRUPTION.
LWT (Lightweight Transaction) = Paxos-based Compare-And-Set (CAS) in Cassandra/ScyllaDB. Write only succeeds IF a condition is true — evaluated atomically. Uses 4-phase Paxos round.
LWT Syntax: 
INSERT INTO users (u_id, u_name) VALUES (12, 'Zurab') IF NOT EXISTS;
UPDATE users SET score = 500 WHERE u_id = 5 IF score = 100.5;
DELETE FROM users WHERE u_id = 5 IF score = 0;
CONSISTENCY SERIAL;  -- Paxos read: sees latest LWT-committed value
SELECT score FROM users WHERE u_id = 5;
LWT Response: Always includes [applied]=True or [applied]=False + current row values.

The 4 Phases of LWT (Paxos-based):
  1. Prepare: Coordinator generates ballot=(timestamp, node_id). Sends PREPARE(ballot) to RF replicas. Gets PROMISE from majority (or REJECT if higher ballot exists).
  2. Read: Coordinator reads current value from quorum to evaluate IF condition. Condition fails → [applied]=False, stop.
  3. Propose (Accept): Coordinator sends ACCEPT(ballot, new_value) to replicas. Quorum accepts → proposal wins.
  4. Commit (Learn): Coordinator sends LEARN(new_value) to replicas. They apply to data table. Coordinator returns [applied]=True to client.
Performance: LWT = 4 round trips vs 1 for normal write → ~4x slower. Only use LWT when the ordering guarantee is truly needed (uniqueness, CAS, financial operations).
LWWLWT
Conflict resolutionLatest timestamp winsCondition must pass (CAS via Paxos)

NEVER mix LWW and LWT writes on the same partition! LWW writes bypass Paxos and can silently overwrite LWT-committed values with a higher timestamp. The LWT ordering guarantee breaks completely. Once you use LWT for a partition, ALL writes to it must go through LWT.
LWT scope: Per-partition, per-operation only. NOT cross-partition. NOT multi-operation transactions.

LWT Error Handling (tricky!): An error (timeout) executing LWT does NOT mean it failed. Paxos may have already committed on some replicas after your client timed out. To confirm: must re-read with CONSISTENCY SERIAL, or retry the operation.

E. Paxos

Auction with multiple auctioneers who can run simultaneously. Only ONE final winner is ever declared, even if some auctioneers disappear mid-auction. Exactly one item gets sold no matter the chaos.
3 Roles: Proposer (initiates, picks ballot) · Acceptor (votes, forms quorum) · Learner (learns the final decision).

Phase 1 — Prepare / Promise:
Proposer picks unique number N → sends Prepare(N) to all Acceptors.
Acceptor replies with Promise(N) IF N > highest N it has promised: "I won't accept lower proposals. Here's what I already accepted (if anything)."
Proposer needs majority of promises.
Phase 2 — Propose / Accept:
Proposer picks value: if any acceptor already accepted a value → MUST use that value. Otherwise free to use own value.
Proposer sends Accept(N, value) to all Acceptors.
Acceptor accepts if N ≥ highest promised N.
Majority accepts → value is CHOSEN. Done.
The critical safety rule: If during Phase 1 you learn a value was already accepted by some acceptor, you MUST carry that value forward in Phase 2. This prevents two Paxos rounds from ever committing different values.
Hard Paxos scenario — 2 competing proposers:
P1 wants value "A". P2 wants value "B". P2 has higher ballot.
P1: Prepare(1) → promises from X,Y. P1: Accept(1,"A") → X accepts. Y,Z reject (promised P2's ballot already).
P2: Prepare(2) → X promises AND reports "I accepted A under ballot 1". Y,Z promise.
P2 must now propose "A" (not "B") because accepted value was found in Phase 1.
Result: consensus on "A" even though P2 wanted "B". Safety guaranteed.
Paxos problems:
  • Livelock: Two proposers keep outbidding each other → no progress. Solution: elect one distinguished leader (like Raft does).
  • Multi-Paxos: Basic Paxos = agree on ONE value. For a log of operations → Multi-Paxos (one Paxos instance per log slot). Raft simplifies this.
SystemHow Paxos used
ScyllaDB/Cassandra LWT4-phase Paxos per LWT operation (Prepare→Read→Propose→Commit)
RaftConceptually Multi-Paxos with strong leader to eliminate livelock
Google SpannerPaxos groups — each shard runs its own Paxos consensus
Apache ZookeeperZAB protocol — heavily Paxos-inspired
Paxos: pick higher number → get promises → propose (must use already-accepted value if found) → majority accepts → done. Raft = Paxos + strong leader + cleaner structure.

Document DB vs Key-Value: Key-value cannot query by value field (DB can't see inside the blob). Document DB uses tagged format (BSON/JSON) → DB understands structure → can filter by any field. That's the critical advantage.
Document DB properties:
  • Queryable by any field (not just key)
  • Flexible (dynamic) schema — different documents in same collection can have different fields
  • Self-contained documents — no FK constraints, no formal relationships
  • Application owns schema responsibility
"What can exist in a MongoDB document that CANNOT exist in a normalized relational table?" → Arrays, embedded sub-documents, varying fields across rows, array of objects with different fields.

B. BSON Structure, ObjectId, Embedding vs Referencing

Hierarchy: Database → Collection (≈Table) → Document (≈Row) → Field (≈Column)
Max document size: 16 MB

ObjectId (12 bytes): 4 bytes Unix timestamp + 5 bytes random (machine+process) + 3 bytes incrementing counter. Globally unique without central coordination. Mandatory _id field.

Embedded Documents

Related data stored INSIDE parent. One read gets everything.
Use when: Always accessed together, 1:1 or 1:few relationship, not independently queried, won't grow unboundedly.

Referenced Documents ($ref)

Related data in separate collection. MANUAL second query needed. No FK enforcement — dangling refs possible.
Use when: Large data, independently queried, write-heavy, unbounded arrays, N:M relationship.
Embedding limitations:
- Full document loaded on every read — even if only 1 field needed
- Update rewrites entire document
- Unbounded array → hits 16MB limit → use references instead
N:M relationships are HARD in MongoDB (no native joins). Must use references + two queries. 1:1 and 1:N are natural with embedding.

C. Single-Leader Replication & Oplog

Single-Leader Replication: ALL writes go to Primary only. Primary sends oplog to Secondaries asynchronously. Secondaries apply changes in oplog order. Reads can go to Primary or a Secondary (depends on Read Preference).
Contrast with P2P (ScyllaDB): P2P: write directly to RF replicas based on quorum. Any node accepts writes. No "primary". Single-leader: only primary accepts writes. Fundamentally different architecture.
Oplog = special capped collection at local.oplog.rs. Records ALL write operations in primary's processing order. Secondaries "tail" the oplog and replay entries.

Oplog is IDEMPOTENT — applying an entry multiple times = applying it once. f(x) = f(f(x)).
Why: $inc: 10 is NOT idempotent (apply twice = +20). Oplog converts to $set: <final_value> instead. Safe to replay on crash recovery.
Why must oplog be idempotent? When a secondary restarts after a crash, it may replay entries it already applied. If they weren't idempotent → data corruption. Idempotent ops make replay always safe.

Read-Your-Own-Write problem: User inserts comment → primary acknowledges → user immediately queries → query routed to secondary that hasn't applied oplog yet → user sees "no results" for their own comment! Solution: use readPreference: primary or w:majority write concern.
Read Preference = WHICH NODE to read from. Read Concern = WHAT VERSION of data to return. They are independent. Understand both.

E. Failover, Split Brain & Rollback

Failover only needed in single-leader systems (MongoDB), NOT in P2P. In P2P, any node accepts writes → no single point of failure. In MongoDB, if primary fails → writes stop until new primary elected.
Failure detection: Heartbeats every 2 seconds. Timeout = 10 seconds (heartbeatTimeoutSecs). If no heartbeat in 10s → member marked inaccessible. Increasing timeout → more accurate (fewer false positives) but slower to detect real failures.

Election process (Raft-based):
  1. Secondary detects missing heartbeat → calls election
  2. Election requires majority of voting nodes available
  3. Candidate with most up-to-date oplog (highest term + commit point) wins
  4. No writes during election period
  5. Old primary rejoins as secondary, may need to rollback uncommitted writes
Split Brain — Scenario 1 (5 nodes, Primary in Europe, 3 Secondaries in US):
Network partition: Europe side = 2 nodes (Primary + Secondary). US side = 3 nodes.
Majority = 3. Europe has 2 → cannot maintain majority → Europe Primary STEPS DOWN.
US has 3 → majority → elects new Primary in US.
Data loss possible: any writes the Europe primary accepted that weren't replicated to US majority → ROLLED BACK.
Split Brain — Scenario 2 (Primary in US, 3 Secondaries in Europe):
US side = Primary + 2 Secondaries = 3 nodes = majority → US Primary stays.
Europe side = 2 nodes only → cannot hold election → becomes read-only.
No split brain. No data loss.
Heartbeat delayed false positive:
US thinks Europe Primary is dead (heartbeat delayed >10s but Primary is actually alive). US elects new Primary → now 2 primaries briefly! Raft term mechanism resolves: new US Primary has higher term. Old Europe Primary, when it reconnects, sees higher term → steps down. Writes to old Europe Primary during this window → ROLLED BACK.
MongoDB Rollback vs PostgreSQL Rollback — key exam question:
MongoDB RollbackPostgreSQL Rollback
CauseAutomatic — triggered by failover/election (replication phenomenon)Intentional — client issues ROLLBACK or transaction fails (transaction phenomenon)
TriggersSystem automatically (old primary when rejoining)Application or DB engine (on error)
Data loss?YES — acknowledged writes (w:1) can be permanently lostNO — transaction was never committed, nothing durable lost
PreventionUse w:majority — majority-acked writes cannot be rolled backN/A — rollback is a feature, not a failure

F. Sharding (Partitioning)

Sharding = splitting data across multiple replica sets. Each subset = a shard. Each shard MUST be a replica set (gets redundancy + scaling together).
Sharded Cluster Architecture — 3 components:
1. Shards — each is a replica set storing a subset of data.
2. Mongos (Router) — client connects here (NOT to shards directly). Routes queries to correct shard(s) using the chunk map. Stateless. Multiple mongos for HA.
3. Config Server (replica set) — stores chunk map (which shard key range → which shard). Mongos caches this. Must also be a replica set.

Range Sharding

Shard key divided into contiguous ranges. Shard 1: 0–30000, Shard 2: 30000–60000...
✓ Range queries efficient (targeted to 1-2 shards)
✗ Hot spots: sequential shard keys (timestamps, auto-IDs) → all inserts to last shard

Hash Sharding

Hash function applied to shard key value. Result determines chunk/shard. Even sequential keys scatter across all shards.
✓ No hot spots, even distribution
✗ Range queries expensive (scatter-gather: sent to ALL shards, results merged)
Chunk: A contiguous range of shard key values assigned to a shard. Mongos knows which chunk is on which shard via the chunk map from Config Server.
Balancer: Background process that moves chunks between shards to maintain even data distribution. Runs automatically.
Use Range sharding when: queries frequently filter by ranges of the shard key. Use Hash sharding when: high-throughput inserts with sequential keys (timestamps, IDs) and no range queries needed.
Scatter-gather vs targeted query: Range sharding with a range query = targeted (mongos sends to 1-2 shards). Hash sharding with any query = scatter-gather (mongos sends to ALL shards and merges results → more expensive).
