1. ACID in P2P Distributed Databases

ACID recap: Databases that support ACID are transactional databases.
Atomicity — all-or-nothing: a sequence of commands runs as a single unit; either all commit or all roll back.
Consistency — all constraints of the database are kept.
Isolation — as if each transaction had the database for itself (serializability).
Durability — writes of committed transactions are persistent, no loss of data.

P2P distributed databases (Cassandra, Scylla, RIAK, ...) treat each ACID letter very differently from a single-server RDBMS:

PropertyStatus in P2P (Scylla/Cassandra)
AtomicityDoes NOT exist in the ACID sense. There is no mechanism that groups a sequence of commands into one unit and commits/rolls them back together. Each CQL statement is executed independently. Cassandra "explicitly chooses not to implement operations that require cross-partition coordination ... does not support cross-partition transactions."
IsolationThere are no transactions, so there are no concurrent transactions to serialize/isolate. But there ARE concurrent operations. Concurrent writes are typically conflicting writes → conflict resolution is needed. Concurrent reads may return different results.
DurabilityOf the four, this is the one P2P databases support most strongly (commit log + replication across nodes).
ConsistencyRedefined entirely (see Section 2). It is not about constraints — it is about replicas agreeing on a value.
A classic exam framing: "Which ACID property is best supported by a P2P distributed database?" → Durability. "Which does not exist?" → Atomicity (no multi-statement / cross-partition transactions).

2. Consistency Means Something Different in P2P

Consistency in P2P = replicas agree on a value. Different replicas return the same value for the same request. This is NOT about database constraints (that is the ACID meaning).

(Atomic) Consistency: The same response is given to all identical requests.

From Kleppmann (p. 322): "If you look at two database nodes at the same moment in time, you're likely to see different data on the two nodes, because write requests arrive on different nodes at different times. These inconsistencies occur no matter what replication method the database uses (single-leader, multi-leader, or leaderless replication)."

Three different "consistency" meanings — do not mix them up:
1. ACID Consistency = all DB constraints are kept (single-server / RDBMS).
2. P2P Consistency = all replicas return the same value for a request.
3. CAP Consistency = every node returns the same, most recent data (clients share one view). The slides explicitly note this is different from the ACID definition.

3. The CAP Theorem

Eric Brewer introduced the CAP theorem in 2000; it was mathematically proven in 2002. It applies to distributed, replicated systems.

LetterPropertyDefinition
CConsistencyClients have the same view of the data. Every node returns the same, most recent data. (Different from ACID consistency!)
AAvailabilityEvery request receives a non-error response in a reasonable amount of time.
PTolerance to network partitionThe system continues to operate and respond correctly even if the network connecting the nodes has a fault.
Key insight — it is NOT really "pick two of three". Network failures (P) are beyond the control of any database system — they simply happen. So a distributed database must tolerate partitions. When a network fault occurs, the real choice is between C and A:
  • C/P (Consistency prioritized): during a partition, refuse requests that cannot guarantee the latest data → may return errors / reduced availability.
  • A/P (Availability prioritized): during a partition, keep answering even if a node returns stale data → tolerate temporary inconsistency.
Scylla / Cassandra / Dynamo-style databases are fundamentally A/P systems (availability + partition tolerance), trading away strong consistency for eventual consistency. The consistency level (Section 8) lets you tune per operation how far toward C you lean.

4. CAP and Eventual Consistency

If a system provides A/P, it tolerates that the database is temporarily in an inconsistent state — concurrent reads to different nodes may return different values.

Eventual Consistency (Kl, p. 322): "Most replicated databases provide at least eventual consistency, which means that if you stop writing to the database and wait for some unspecified length of time, then eventually all read requests will return the same value. In other words, the inconsistency is temporary, and it eventually resolves itself."

Eventual consistency is a weak consistency guarantee: → eventually, a read will return the same result on all nodes.

It is used for applications that:

  • demand high availability (even with network partitions or node outages),
  • need low latency,
  • can tolerate stale reads and write-conflict resolution.

ScyllaDB, Cassandra and other NoSQL databases "sacrifice a degree of consistency in order to increase availability. Rather than providing strong consistency, they provide eventual consistency. This means that in some cases, a read request will fail to return the result of the latest WRITE."

5. Why Two Replicas Hold Different Data

The exam asks directly: "How can it happen that 2 replicas have different data for the same object?" There are exactly two root causes:

  1. Replication / write propagation delay (replication lag): A write reaches the coordinator and is forwarded to the replica nodes, but not all replicas have applied the latest write yet. For a short window, some replicas hold the new value and some still hold the old one.
  2. Concurrent writes: Two clients write the same object at the same time using different coordinators. The replicas end up with genuinely different values, and the system must resolve the conflict (Section 16 onwards).
Client 1: update users1 set activity_score = 10 where u_id = 11; (coordinator n10) Client 2: update users1 set activity_score = 50 where u_id = 11; (coordinator n7) --- concurrent writes, no natural order ---
These are the two scenarios you must be able to name. Cause 1 (propagation delay) resolves itself once replication completes. Cause 2 (concurrent writes) requires an explicit conflict-resolution rule because the writes have no natural order.

6. Happened-Before and Concurrency

Happened-before (Kl, p. 187): "An operation A happens before another operation B if B knows about A, or depends on A. ... Two operations are concurrent if neither happens before the other (i.e., neither knows about the other)."

For any two operations A and B there are exactly three possibilities: A happened before B, B happened before A, or A and B are concurrent.

Consequences for Cassandra / Scylla / Dynamo-style systems:

  • Concurrent writes do not have a natural order.
  • If an order existed, the system would simply apply the writes in that order → no data loss, no lost updates.
  • Because there is no given order, concurrent writes cause a conflict that the system must resolve by some rule. The system needs a mechanism to decide which value "wins".

7. Two Causes of Inconsistency & Their Control Mechanisms

This table directly maps each cause of CAP-style inconsistency to the mechanism that drives the system back toward eventual consistency:

Concurrent writesReplicas have different writesSystem conflict-resolution rules (LWW, version vectors, siblings, CRDTs, consensus)

8. Consistency Levels (one / quorum / all)

The consistency level determines how many replicas must participate in a read or write operation.
LevelMeaning
consistency oneone replica needs to participate
consistency quorumthe majority of replicas need to participate
consistency allall replicas must participate

Default consistency level in Scylla / Cassandra: one.

"Quorum = majority". For replication factor RF, quorum = ⌊RF/2⌋ + 1. For RF=3, quorum = 2. For RF=5, quorum = 3. The default level one is the fastest and most available, but the most likely to return stale data.

9. Scylla Consistency Concept — Quorum Details

  • Write quorum (w quorum): a majority of replicas must apply the write.
  • Read quorum (r quorum): a majority of replicas must respond to a read request.
  • If a write or read does not reach quorum, it is considered failed and returns an error.
  • The coordinator node is responsible for returning the acknowledgement / failure to the client.
Critical edge case (Kl, p. 178): If a quorum write times out before the majority of replicas can acknowledge to the coordinator, the coordinator sends a failure back to the client — but some replicas may have already applied the write, and eventually that write may become persistent. Applied writes are NOT rolled back. So a "failed" quorum write can still leave data behind.
This edge case is the P2P parallel of the LWT "timeout ≠ failure" rule from Module 6.4: a reported failure does not guarantee the write was not applied. There is no rollback in P2P.

10. Watching Consistency Behavior (Lab Exercises + Answers)

Setup: rf = 3; u_id 14 is stored on kiu-scylla2 and kiu-scylla3 (verified via nodetool getendpoints); node status via nodetool status.

Exercise A — consistency one, stale read after node restart

  1. Default consistency = one. Stop kiu-scylla3.
  2. On kiu-scylla2: read activity_score of user 14, modify it in the app, write back the updated score.
  3. The update is acknowledged (only one replica needed). Confirm the new value shows.
  4. Now stop kiu-scylla2 and start kiu-scylla3; open cqlsh on kiu-scylla3.
  5. Q: Which value does kiu-scylla3 show? Why?

Answer: kiu-scylla3 shows the OLD (stale) value. With consistency one the write only had to be applied on a single replica (kiu-scylla2). kiu-scylla3 was down when the write happened, so it never received the update, and now kiu-scylla2 (which holds the new value) is down — so read repair / replication cannot fix it yet. This demonstrates eventual consistency: until replication/repair catches up, replicas disagree.

Exercise B — consistency quorum, write availability under a downed node

  1. Set consistency = quorum on the session doing the update (kiu-scylla2). Stop kiu-scylla3.
  2. Try to write user 14 (its replicas include the stopped node).
    Q: How does Scylla react and why?

Answer (user 14): Quorum for RF=3 is 2. If two of user 14's three replicas are still up, the write succeeds (quorum reached). If the down node leaves fewer than 2 replicas of user 14 reachable, the write fails with an error because quorum cannot be met.

  1. In the same quorum session, write back an updated activity_score for user 5.
    Q: How does Scylla react and why?

Answer (user 5): Depends on whether the down node holds a replica of user 5. If user 5's replicas are all on running nodes, the quorum write succeeds; if the stopped node was needed to reach the majority, it fails. The point: quorum availability is per object, decided by which replicas are up.

  1. Run select * from users from a quorum session and from a non-quorum (one) session.
    Answer: The quorum read may return a fresher / more consistent result (and may fail if quorum is unreachable); the one read always returns quickly from a single replica but can be stale.

11. P2P Consistency Diagrams (Write & Read Scenarios)

Ring of 10 nodes, rf = 3, n10 is the coordinator. The coordinator forwards the operation to the RF replicas.

Write with consistency quorum, one replica unreachable

Client 1: insert into users values (John, 20, ...) -> coordinator n10 n10 forwards write to replicas n2, n3, n4 n2: John,20 (applied) n3: John,20 (applied) n4: UNREACHABLE Write quorum (2 of 3) reached on n2 + n3 -> WRITE SUCCEEDS

The write is acknowledged because 2 of 3 replicas applied it; n4 will catch up later via repair.

Read with consistency one (can return stale 0)

Client 1: insert (John, 20) just happened, still propagating Client 2: read(John) -> coordinator n7 read consistency = ONE n4 running but read executes there -> returns 0 (write not yet applied on n4!) n2 read delayed (holds John,20) n3 read delayed (holds John,20) Result: client may get "0" -> STALE READ

Read with consistency quorum (waits for majority, may time out)

Client 2: read(John) -> coordinator n7, read consistency = QUORUM needs majority (2 of 3) to respond n4 returns 0 ; n3 returns John,20 -> quorum responded The read SUCCEEDS but the two responding replicas hold DIFFERENT values -> triggers read repair (Section 12) to reach eventual consistency (If too few replicas respond before timeout -> read FAILS with error)
Pattern to memorize: consistency one = fast, always answers, can be stale. consistency quorum = waits for a majority, more likely fresh, but fails on timeout if the majority cannot respond. A successful quorum read that sees disagreeing replicas triggers read repair.

12. Read Repair

Read repair: a read operation triggers a write operation to synchronize replica values. It is one of the operational control mechanisms that drives the cluster toward eventual consistency.

How it works in Scylla / Cassandra:

  1. A successful read detects different values for the requested data across replicas.
  2. The coordinator takes the value with the latest timestamp and returns it to the client.
  3. It then starts a write process (the read-repair) that writes the latest-timestamp version back to the out-of-date replicas.

RIAK describes the same idea with vector clocks: read repair occurs when a successful read finds that not all replicas agree. The errant node either responded "not found" (no copy) or with a vector clock that is an ancestor of the successful read's vector clock. RIAK then forces the errant nodes to update to the value of the successful read.

13. Write/Read Quorum Replication — Worked Examples

Notation: N = number of replicas, W = replicas that must participate in a write, R = replicas that must participate in a read. With N = 5, quorum = 3.

Two diagnostic questions for each setting:
(a) How many nodes can be down while the cluster keeps write availability? → N − W
(b) Can you get stale reads only? → only when W + R ≤ N (no guaranteed overlap)

ExampleN / W / RNodes that can be down (write still available) = N − WStale reads possible? (W + R vs N)
1N=5, W=quorum(3), R=one(1)5 − 3 = 2W+R = 4 ≤ 5 → YES, stale reads possible (no guaranteed overlap)
2N=5, W=one(1), R=quorum(3)5 − 1 = 4W+R = 4 ≤ 5 → YES, stale reads possible
3N=5, W=quorum(3), R=quorum(3)5 − 3 = 2W+R = 6 > 5 → NO — read always overlaps a fresh replica → returns most recent
Example 3 (W=quorum, R=quorum) is the sweet spot: W+R > N guarantees a read overlaps at least one node that has the latest write, so you cannot get only-stale reads — at the cost of needing a majority for both reads and writes. Example 2 maximises write availability (4 nodes can be down) but R=quorum makes reads heavier and still allows stale results.

14. The W + R > N Rule

W < N and R < N → allows the system to continue processing when nodes are down (you never need all replicas).

W + R > N → a read returns — from at least one node — the most recent value for an object, because the write set and the read set are guaranteed to overlap in at least one replica.

Intuition: if writes always land on W replicas and reads always touch R replicas, and W + R exceeds the total N, the two sets cannot be disjoint — at least one read replica must be one of the freshly written ones. Consistency levels are the operational control that lets you choose W and R, and therefore tune the consistency/availability trade-off per operation.

15. Read-Modify-Write (No Delta Updates)

With Cassandra / Scylla you must do the read-modify-write cycle in the application. You cannot issue a delta update like "add 10 to the current value" — the database does not read-then-add atomically for you.

Example: users table, activity_score starts at 50

App 1: give a 10% benefit → activity_score × 1.1 (= 55)
App 2: increase by 10 → activity_score + 10 (= 60)

Both run concurrently, unaware of each other, on different coordinators. Each does: SELECT activity_score (both read 50), compute locally, then UPDATE ... SET activity_score = ....

Client 1: select activity_score where u_id=14 -> 50 ; update set activity_score = 55 Client 2: select activity_score where u_id=14 -> 50 ; update set activity_score = 60 --- concurrent writes, both based on the stale value 50 --- Correct combined result should be (50*1.1)+10 = 65, OR (50+10)*1.1 = 66. Actual result with LWW: either 55 or 60 survives -> the other update is LOST.

Lost update. Because both read 50 before either wrote, one update silently overwrites the other. Neither 65 nor 66 is reached. This is the motivation for Lightweight Transactions / consensus (Module 6.3/6.4) when you truly need correctness.

16. Write-Write Conflict Solution Strategies (Overview)

When concurrent writes conflict, a P2P system needs a rule to decide what to keep. The five strategies (covered in detail in Part 6.2):

#StrategySystemsIdea
1LWW (Last Write Wins)Scylla (default), Cassandra (default), RIAK (optional), Redis & CouchDB (some config), Apache Kafka, Azure Cosmos DBArbitrary order imposed by the system (via timestamps) to resolve the conflict.
2Version VectorsRIAKDetect happened-before / after / concurrent. Detect conflicts, do not resolve them.
3Sibling ValuesRIAKKeep both conflicting values; the application decides.
4CRDTsRIAK, RedisData structures that merge automatically. Concurrency: yes, conflicts: no.
5Consensus MechanismCassandra, ScyllaEnforce an agreed-upon order, preventing conflicts from concurrent writes (LWT / Paxos — Module 6.4).

17. LWW (Last Write Wins) — In Depth

LWW core idea: Concurrent writes have no natural order, so the system imposes an arbitrary order, usually decided via generated timestamps. If two concurrent writes hit the same object, the write with the highest timestamp survives.
  • Scylla / Cassandra: timestamp generation is NOT coordinated cluster-wide. Each coordinator generates timestamps from its own system clock.
  • → Clock drift / skew between nodes is a fundamental problem in LWW. A node whose clock runs ahead will always win (cf. Kl ch. 8, unreliable clocks).
  • If two timestamps are equal, the system uses some tie breaker, which is usually not well documented.
LWW trades durability for convergence (Kl, p. 186): "LWW achieves the goal of eventual convergence, but at the cost of durability: if there are several concurrent writes to the same key, even if they were all reported as successful to the client (because they were written to w replicas), only one of the writes will survive and the others will be silently discarded."
Exam trap: "All concurrent writes were acknowledged as successful — is data safe?" → No. With LWW, acknowledgement does not mean survival; all but the highest-timestamp write are silently dropped. This is why LWW is fine for caches/telemetry but dangerous for data you cannot afford to lose.

18. LWW Lab Exercises (writetime, USING TIMESTAMP)

Exercise A — watch a write get silently discarded

-- 2. read the current name of u_id 10
SELECT u_name FROM users1 WHERE u_id = 10;

-- 3. read the cell's write timestamp
SELECT writetime(u_name) FROM users1 WHERE u_id = 10;
-- returns a large microsecond timestamp, e.g. 1718000000000000

-- 4. update with a LOWER (older) timestamp
UPDATE users1 USING TIMESTAMP 1000 SET u_name = 'AAA' WHERE u_id = 10;

Q: Will the update run? What do we expect?

Answer: The command executes without error (it is accepted), but because 1000 is far lower than the existing cell's timestamp, LWW keeps the existing (higher-timestamp) value. Step 5 check: u_name is still the original value, not 'AAA' — the write was silently discarded. This proves LWW compares timestamps, not arrival order, and that a "successful" command can have no effect.

Exercise B — concurrent update vs delete, decided by timestamp

Two concurrent writes on u_id = 10: one updates the activity score, the other deletes the row. By clock drift the update carries a timestamp slightly higher than the delete.

-- reproduce deterministically:
SELECT writetime(activity_score) FROM users1 WHERE u_id = 10;   -- find current TS
-- run the DELETE with TS = current + small amount
DELETE FROM users1 USING TIMESTAMP <ts+1> WHERE u_id = 10;
-- run the UPDATE with a yet slightly higher TS
UPDATE users1 USING TIMESTAMP <ts+2> SET activity_score = 99 WHERE u_id = 10;

Q: What is the result? Does the order of actual application matter?

Answer: The write with the highest timestamp wins regardless of the physical order in which the replicas apply them. Here the update (ts+2) beats the delete (ts+1), so the row exists with activity_score = 99. It does not matter which command physically arrives first — LWW is decided purely by timestamp. (Note: a delete actually writes a tombstone with its timestamp; whichever timestamp is higher — tombstone or update — wins.)

19. Version Vectors / Vector Clocks — Concept

A version vector (VV) lets participants determine, for two updates, whether one preceded the other (happened-before), followed it (happened-after), or whether they happened concurrently. VVs version values/updates — they are a "logical clock".
TermPurpose
Version vectorsA mechanism to synchronize replicas.
Vector clocksA mechanism for partial (causal) ordering of events in a messaging system (e.g., so you don't get the answer before the question).

Both are vector counters. They follow slightly different algorithms and methods, but are often used as synonyms.

20. Version Vector Mechanics

  • A version counter is kept per replica, per object.
  • Each replica increments its own counter when it processes a write on an object.
  • A version vector is the collection of all replicas' version counters for that object. Its size = the replication factor.
  • When two replicas synchronize, they set each slot to the maximum of the two:
V1[x] = V2[x] = max( V1[x], V2[x] )    for every replica slot x
-- after synchronization, both replicas hold identical version vectors
-- (this works when the two vectors are COMPARABLE)
Comparing two VVs (componentwise):
• VV_A ≤ VV_B in every slot, and strictly smaller in at least one → A happened-before B (B is more recent; A is an ancestor).
• Symmetric case → B happened-before A.
• Neither dominates (one slot bigger here, another slot bigger there) → the vectors are incomparable → the writes are concurrent → a conflict.

21. VV Write, Read & Read-Repair Walkthrough

Replicas n2, n3, n4 (rf=3). Stored data structure: {key, value, VersionVector}. Start: (John, 5, (n2:1,n3:1,n4:1)).

Write operation (client A updates John → 20; n4 is down)

client A: update(John, 20) -> Coordinator n2 applies, increments its slot: (John, 20, (n2:2,n3:1,n4:1)) n3 applies, increments its slot: (John, 20, (n2:2,n3:2,n4:1)) n4 DOWN -> still (John, 5, (n2:1,n3:1,n4:1))

Note: each writing replica bumps its own slot. Both LWW and VV attach metadata per replica/object — but LWW attaches a single timestamp, whereas VV attaches the whole counter vector.

Read + read-repair (client B reads John, reading from n2 and n4)

n2: (John, 20, (n2:2,n3:2,n4:1)) n4: (John, 5, (n2:1,n3:1,n4:1)) Compare: (n2:1,n3:1,n4:1) <= (n2:2,n3:2,n4:1) in every slot, strictly smaller -> n4's VV is STRICTLY SMALLER -> n4 happened-before n2 -> n2 is more recent Therefore: -> return n2's value (John, 20) to the client -> update (repair) n4 to (John, 20, (n2:2,n3:2,n4:1)) -> both nodes now hold the same VV -> synchronized
When VVs are comparable, there is no real conflict — the larger vector simply wins and the smaller replica is repaired (synced to the max). The interesting case is when they are incomparable (next section).

22. Concurrent Writes with VV — Incomparable Vectors

Two concurrent updates from two clients (RIAK)

Start: (John, 5, (n2:1,n3:1,n4:1)) on all replicas.

client 1: update(John, 20) -> produces (John, 20, (n2:2,n3:1,n4:1)) then (n2:2,n3:2,n4:1) client 2: update(John, 50) -> produces (John, 50, (n2:1,n3:1,n4:2)) then (n2:1,n3:2,n4:2)

Q: Are the VVs comparable?

Answer: No. Compare (n2:2,n3:2,n4:1) vs (n2:1,n3:2,n4:2): the first is larger in the n2 slot (2>1) but smaller in the n4 slot (1<2). Neither dominates → the vectors are incomparable → the two writes are concurrent → genuine conflict.

Q: What is returned to the clients? What happens to the VVs?

Answer: The VV mechanism detects the concurrent conflict but cannot resolve it on its own. What happens next depends on the system / setting: RIAK with allow_mult(true) keeps both values as siblings (Section 24); otherwise a resolution policy (e.g., LWW) picks one. The conflicting VVs are kept until the application resolves the conflict.

23. Detect vs Resolve — The Key VV Exam Question

Slide multiple-choice — Version Vectors:
A) are a method to resolve concurrent write conflicts
B) detect and resolve concurrent write conflicts
C) can only detect concurrent write conflicts

Correct answer: C. Version vectors can only DETECT concurrent write conflicts. They do not resolve them — resolution depends on the system and/or configuration (siblings, application logic, LWW, CRDTs, etc.).

24. Sibling Values (RIAK)

Siblings: when a conflict is detected and RIAK is configured with .allow_mult(true), both conflicting values are kept as sibling values.
  • → Both values are returned and both VVs are kept.
  • → The client / application must resolve the conflict and decide on one value.
  • → After resolution, the chosen value and its VV are kept.

RIAK documentation: "Concurrent writes — If two writes occur simultaneously from clients, Riak may not be able to choose a single value to store, in which case the object will be given a sibling."

25. Advantage of Version Vectors

VVs detect concurrent writes but cannot resolve them — so what is their advantage over plain LWW timestamps?

VVs capture "happened-before", "happened-after" or "happened-concurrently" — i.e. they encode the causal history of an object. A plain timestamp throws this information away. 
node A: [A:3, B:2]
node B: [A:2, B:2]
-- A dominates B (3>=2 and 2>=2, strictly greater in slot A)
-- => the VV on node A includes ALL writes applied by node B, plus one extra write at A
-- => no conflict; A is simply more recent

By contrast, a single timestamp imposes an arbitrary order. A higher timestamp may:

  • truly represent a happened-after relation, OR
  • actually correspond to a happened-before relation because of clock skew, OR
  • represent a concurrent write that "won" only because of an arbitrary tie breaker.
The advantage of VVs: they can tell the difference between "genuinely newer" and "concurrent / conflicting". LWW timestamps cannot — they always declare a winner even when the writes were really concurrent, silently losing data.

26. CRDTs — Conflict-Free Replicated Data Types

CRDT = Conflict-free Replicated Data Type. Also expanded as convergent replicated data types (RIAK) and commutative replicated data types (Redis). Formally defined in 2011 by Shapiro, Preguiça, Baquero & Zawirski. CRDTs are data structures that fulfil certain conditions and perform the merge automatically following the data type's inherent merge rules.

Wikipedia definition — a CRDT is a data structure replicated across multiple computers with:

  • The application can update any replica independently, concurrently, and without coordinating with other replicas.
  • An algorithm (part of the data type) automatically resolves any inconsistencies.
  • Replicas may differ at any instant, but are guaranteed to eventually converge.

RIAK's motivation: conflict resolution is hard because it involves reasoning about concurrency, eventual consistency and siblings. CRDTs move that burden from the application into the database — "Without using data types, that conflict must be resolved using timestamps, vector clocks, dotted version vectors, or some other means. With data types, conflicts are resolved by Riak KV itself."

Slogan from the strategy table: CRDTs give you concurrency: yes, conflicts: no. The key trick (next section) is to reduce all writes to commutative operations so the order of writes no longer matters.

27. CRDT Grow-Only Counter (Worked Example)

Nodes n1, n2, n3. Each holds a state vector of per-node counts; each node may only update its own slot. The merge function takes the maximum of each slot and is commutative. The counter's current value = sum of all slots.

start: n1:(0,0,0) n2:(0,0,0) n3:(0,0,0) client A: inc c := c+2 on n1 -> n1:(2,0,0) client B: inc c := c+3 on n2 -> n2:(0,3,0) merge n1 & n2 (max per slot) -> (2,3,0) on both merge into n3 (was down) -> n3:(2,3,0) final value = sum(2,3,0) = 5 (correct: 2 + 3, NO lost update)
Why no lost update? Because each node only ever increments its own slot, two concurrent increments land in different slots and the max-merge keeps both. Compare this with the read-modify-write lost-update problem of Section 15 — the CRDT counter would have correctly produced 5, whereas LWW would have kept only one increment.

28. State-Based vs Operation-Based CRDTs

State-based CRDT: consists of a state, application algorithms, and a replication protocol. The merge method is the most important operation and must satisfy three properties:
  1. Commutative — merge(a,b) = merge(b,a)
  2. Associative — merge(merge(a,b),c) = merge(a,merge(b,c))
  3. Idempotent — merge(a,a) = a
The replication protocol only needs to guarantee that eventually all state updates are delivered to all replica nodes.
State-based CRDTOperation-based CRDT
What is sentThe client sends the new state; it merges its state with other nodes' states (does not pass on the operations).The update operation is propagated and applied locally.
Delivery requirementLoose — eventual delivery is enough (merge is idempotent, so duplicates are harmless).Stricter — each operation must be transmitted exactly once per replica, because in some formulations operations are not idempotent.
OrderingCan be merged in any order (commutative + associative).Once delivered, can be applied in any order.

29. Strong Eventual Consistency (SEC)

Strong Eventual Consistency (SEC): an object satisfies SEC if all correct replicas that have delivered the same set of updates have equivalent state — without needing conflict resolution or rollbacks.

Theorem: A state-based CRDT satisfies SEC. (Proof by induction on the causal histories of deliveries at the replicas — INRIA research report RR 7687.)

SEC is stronger than plain eventual consistency: ordinary eventual consistency only promises convergence "eventually after writes stop"; SEC guarantees that any two replicas that have seen the same updates are already in the same state, with no conflict resolution needed — which is exactly what CRDTs deliver.
