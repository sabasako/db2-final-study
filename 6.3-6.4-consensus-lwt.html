1. What is Consensus?

Consensus Definition (from slides, [Kl] p.364):
A distributed system (database) guarantees a result — a value, an order, or a state — despite failures, even though the agreement may have been reached by a majority quorum only rather than by all nodes directly.

One or more nodes propose a result, and the consensus algorithm decides on one of those results.

All nodes agree on the result despite possible failures, including:

  • Node crashes
  • Asynchronous communication and replication delays
  • Lack of cluster-wide clock coordination (no global clock)
  • Network partitions (some nodes can't reach others)

Only a majority quorum — not necessarily all nodes — needs to be involved in reaching the agreement. This makes consensus fault-tolerant: as long as a majority is alive, consensus can be reached.

What Can Be "Agreed Upon"?

  • A specific value — for data consistency (e.g., "the new credit balance is 500")
  • Cluster definition and membership — which nodes are in the cluster, which have left or joined
  • Cluster metadata — schema definitions, topology information
  • The outcome of a distributed transaction — commit or abort
  • A cluster-wide order of events — establishing which event happened first
Key insight: Consensus is a core concept of distributed systems and distributed databases. Without it, nodes might disagree on the current state, leading to inconsistency.

2. Consensus Properties

A correct consensus algorithm must satisfy four formal properties:

PropertyDefinitionIn Practice
AgreementNo two correct nodes decide on different valuesAll nodes that decide must decide the same thing — no split decisions
IntegrityNo node decides twiceOnce a node has decided on a value, it does not change its decision
ValidityAny decided value was proposed by some nodeThe decided value was not invented out of thin air — it was a real proposal
TerminationEach node that does not crash eventually decidesThe algorithm makes progress — it does not loop forever
Agreement + Validity together mean: all correct nodes agree on a value that was actually proposed. Termination means the algorithm doesn't stall forever. These four properties define what it means to "solve" consensus.

Why Consensus is Hard

The famous FLP Impossibility Result (Fischer, Lynch, Paterson, 1985) proves that no deterministic consensus algorithm can guarantee both safety (agreement + validity) and liveness (termination) in a fully asynchronous network where even one process can crash. In practice, systems get around this by:

  • Assuming partial synchrony (messages eventually arrive)
  • Using timeouts to detect failures (though imperfect)
  • Accepting that in very rare cases the algorithm may need extra rounds

4. NoSQL Databases and Consensus

Many NoSQL distributed databases (Cassandra, ScyllaDB, MongoDB, Kafka, ElasticSearch...) initially avoided consensus protocols because they were considered too costly — too many round trips, too much coordination overhead. The optimistic approach was that temporary inconsistencies could be tolerated.

In recent years, most of these systems have introduced consensus-based solutions selectively for:

  • Topology issues — tracking which nodes are in the cluster
  • Cluster coordination — coordinating schema changes across nodes
  • Leader election — safely electing a new primary after failure
  • Option to enforce invariants from the application side — LWT in ScyllaDB
Operations that manage the cluster itself (topology, schema) rely more and more on consensus protocols. Operations on user data (write path / read path) still largely avoid consensus for performance. But even for user data, there are now options to enforce consensus — at the cost of performance.

The two key protocols used today:

  • Paxos family — classic consensus, used for LWT in ScyllaDB/Cassandra
  • Raft — designed for understandability, used for leader election (MongoDB) and cluster management (ScyllaDB)

5. Two-Phase Commit (2PC) Protocol

2PC Purpose: A protocol that ensures all databases involved in a distributed transaction either ALL commit or ALL abort. Without this, some DBs could commit while others roll back, leaving the system inconsistent.
Do NOT confuse 2-Phase Commit (2PC) with 2-Phase Locking (2PL)!
2PL = a locking protocol within a single database for concurrency control.
2PC = a coordination protocol across multiple databases for distributed commits.

The Problem 2PC Solves

Imagine a bank transfer: debit from Postgres, credit to Oracle. If Postgres commits but Oracle crashes before committing, money disappears. We need atomic decision across both systems.

2PC Protocol — Step by Step

Setup: One node acts as the Coordinator (transaction manager). All other databases are participants (cohorts).

Phase 0 (Pre-phase): Execution

  • Coordinator generates a globally unique transaction ID
  • All operations carry this transaction ID
  • Operations execute on each database; if any fails, transaction aborts immediately

Phase 1 (Prepare/Voting Phase):

  1. Coordinator sends PREPARE (or "prepare for commit") to all participants
  2. Each participant checks if it can commit: all constraints satisfied? WAL flushed?
  3. Each participant votes YES (can commit, will not unilaterally abort) or NO (cannot commit)
  4. Coordinator waits until ALL participants respond (or times out)
  5. A YES vote is a binding promise — the participant must commit when told to, even if it crashes and recovers

Phase 2 (Commit/Abort Phase):

  1. If ALL voted YES: coordinator writes "COMMIT" to its transaction log on disk (this is the commit point)
  2. Coordinator sends COMMIT to all participants
  3. Participants commit and send acknowledgment
  4. If ANY voted NO (or timed out): coordinator sends ABORT to all participants
  5. Participants roll back
  6. If commit fails on any node, it is retried until it succeeds (binding vote!)

6. XA Standard for Distributed Transactions

The XA Standard (eXtended Architecture) is a standardized protocol for 2PC that allows transactions to span multiple heterogeneous database systems. It is implemented by PostgreSQL, MySQL, MariaDB, DB2, SQL Server, Oracle, and others.

XA Architecture

A Transaction Manager (coordinator) sits above multiple databases and orchestrates the 2PC using XA commands:

XA START 'transaction_id';     -- Begin transaction on a participant
... SQL operations ...
XA END 'transaction_id';       -- Mark end of SQL work
XA PREPARE 'transaction_id';   -- Phase 1: vote YES or NO
XA COMMIT 'transaction_id';    -- Phase 2: commit (if all voted YES)
XA ROLLBACK 'transaction_id';  -- Phase 2: abort

How XA Works in Practice

The coordinator uses XA commands to manage each participant database independently. The coordinator itself keeps a transaction log recording the final decision. This log survives crashes, ensuring the coordinator can recover and re-send COMMIT or ABORT commands after restart.

In PostgreSQL: PREPARE TRANSACTION 'txid'; is the Phase 1 vote. The prepared transaction persists across crashes. Then COMMIT PREPARED 'txid'; or ROLLBACK PREPARED 'txid'; completes Phase 2.

7. 2PC Failure Scenarios — Cohort (Participant) Failures

During Prepare Phase:

  • One database votes "abort": Coordinator receives the NO vote → immediately decides ABORT → sends ABORT to all participants → all roll back. Result: clean abort, system consistent.
  • One database becomes unavailable before sending its vote: Coordinator times out waiting → treats missing vote as NO → decides ABORT → sends ABORT to all. The unavailable node, when it recovers, either rolls back (it never voted YES) or queries the coordinator for the decision. Result: abort.

During Commit Phase:

  • One database becomes unavailable after sending its YES vote: That node already voted YES (binding promise) and then crashed. Coordinator still gets YES from others. Coordinator still decides to COMMIT (if all others said YES). The unavailable node, on recovery, queries the coordinator and then commits. The coordinator retries the COMMIT to that node until it succeeds. Result: eventual commit on all nodes.

8. 2PC Failure Scenarios — Coordinator Failures

During Prepare Phase:

Coordinator sends PREPARE requests to participants, then crashes:

  • At least one participant has not sent its vote yet → it is safe for that participant to abort (it never committed to anything)
  • Participants don't know whether coordinator wrote the commit decision to its log → they don't know the final outcome
  • Participants are in a state of doubt (they voted YES but haven't received commit/abort)
  • They are blocked — holding locks — until the coordinator recovers (this is why 2PC is called a "blocking protocol")
  • When coordinator recovers: it reads its log. If PREPARE was logged but no decision → abort. If COMMIT was logged → send COMMIT to all.

During Commit Phase:

Coordinator made and logged the COMMIT decision, started sending COMMIT, then crashes:

  • Some participants received COMMIT, some didn't
  • Participants communicate among themselves: if any participant received a COMMIT or ABORT, it is safe for others to follow that decision
  • Result: participants can eventually determine the outcome by querying each other
The critical weakness of 2PC: if the coordinator crashes AFTER all participants voted YES but BEFORE writing the decision to its log, participants are permanently blocked (holding locks) until coordinator recovers. This is the "blocking problem" of 2PC.

9. 2PC True/False Analysis (from slides)

StatementAnswerExplanation
A. Coordinator is the single point of failure. TRUE If the coordinator fails after participants vote YES but before sending COMMIT, participants are blocked indefinitely. The coordinator is the bottleneck for the final decision.
B. Coordinator is the single point for the final commit/abort decision. TRUE Only the coordinator makes and records the final COMMIT or ABORT decision. Participants cannot make this decision themselves.
C. Any involved node can block 2PC when crashing before voting "Yes" in the prepare phase. TRUE If a participant crashes without voting, the coordinator times out and aborts. This blocks completion (though in this case, it aborts rather than hangs indefinitely).
D. Any involved node can block 2PC when crashing after voting "Yes" in the prepare phase. FALSE If a participant crashes AFTER voting YES, it is committed to voting YES. The coordinator can proceed with the other participants' votes. If it still gets a quorum of YES votes, it commits. The crashed participant, on recovery, gets the COMMIT instruction and commits. 2PC is NOT blocked in this case.

10. Alternatives to 2PC

2PC is a blocking protocol — if the coordinator fails after the vote, participants are stuck holding locks. This motivated alternatives:

1. Three-Phase Commit (3PC)

  • Adds a third phase: "pre-commit" between PREPARE and COMMIT
  • In the pre-commit phase, participants know the coordinator intends to commit
  • If coordinator fails, participants can complete the commit themselves
  • Requires synchronous communication (reliable message delivery with timeouts)
  • In practice: essentially not implemented in commercial databases — too complex, synchronous communication assumption too strong in real networks

2. 2PC with a Full Consensus Algorithm (e.g., Paxos)

  • Every node broadcasts its prepared vote to all other nodes
  • Each node counts the votes it receives
  • If any node receives an ABORT vote → transaction can immediately be aborted
  • Otherwise, a node waits until it has received one vote from each node
  • If ALL votes are YES → transaction can be committed
  • Basic idea: all nodes agree on a decision (consensus) and each makes the same decision independently
  • No single coordinator → no single point of failure
  • More round trips but more fault tolerant
The key difference: in classic 2PC, only the coordinator decides. In Paxos-based 2PC, all nodes independently reach the same decision through consensus. This eliminates coordinator as single point of failure.

11. Raft Consensus Algorithm

Raft is a consensus algorithm designed to be easier to understand than Paxos while providing equivalent guarantees. It is used by ScyllaDB for cluster management and by MongoDB for leader election.

Raft Core Concepts

  • Term: Raft divides time into numbered terms (1, 2, 3...). Each term begins with an election. Terms act as logical clocks — if a node sees a message with a higher term, it updates its term.
  • Roles: Each node is one of: Follower, Candidate, Leader
  • Log: An ordered sequence of entries (operations). Leader appends, followers replicate.

Leader Election in Raft

  1. All nodes start as Followers
  2. Followers wait for heartbeats from the Leader. If none arrives within the election timeout (typically 150-300ms), the follower assumes the leader is dead
  3. Follower becomes a Candidate, increments term, votes for itself
  4. Candidate sends RequestVote RPC to all other nodes
  5. A node grants a vote if: the candidate's term >= own term AND the candidate's log is at least as up-to-date as the voter's log
  6. First candidate to receive votes from a majority of the cluster wins and becomes Leader
  7. Leader sends heartbeats (AppendEntries with no entries) to all followers to maintain authority

Split Vote Scenario

Two candidates start election simultaneously and each gets half the votes (no majority). Solution: each node uses a randomized election timeout so one usually starts first. If split vote occurs, the term increments and a new election starts.

Log Replication in Raft

  1. Client sends command to Leader
  2. Leader appends command to its log (as an uncommitted entry)
  3. Leader sends AppendEntries RPC to all Followers
  4. When majority of cluster acknowledge the entry → Leader marks it committed
  5. Leader applies command to state machine (executes it)
  6. Leader notifies followers in next heartbeat, followers commit too

Safety Guarantees of Raft

  • Election safety: At most one leader per term
  • Log matching: If two logs have an entry with the same index and term, they are identical up to that point
  • Leader completeness: A committed entry from term N will be present in all leaders of term >N

How ScyllaDB Uses Raft

ScyllaDB uses Raft for:

  • Cluster management: Nodes joining/leaving, topology changes
  • Schema management: DDL operations (CREATE TABLE, DROP TABLE, ALTER TABLE) are applied via Raft so all nodes see the same schema

This gives reliable, consistent cluster operation without the old gossip-based approach which could lead to schema disagreements between nodes.

How MongoDB Uses Raft (Replica Set Elections)

MongoDB Replica Set elections are Raft-based:

  • When Primary fails → Secondaries detect missing heartbeat
  • Secondary starts election, sends replSetRequestVotes
  • Node with most up-to-date oplog (highest term + highest commit point) wins
  • New Primary elected; old Primary becomes Secondary when it recovers

12. Consensus-Based Mechanisms in ScyllaDB

ScyllaDB started out avoiding consensus but over time introduced it in two areas:

1. Raft for Cluster and Schema Management

Motivation: Reliably working cluster (members leaving and joining, partitioning) and reliable database schema for nodes to work with. Previously: gossip-based membership could lead to inconsistencies. Now: Raft ensures all nodes agree on membership and schema.

2. Paxos for Lightweight Transactions (LWT)

ScyllaDB and Cassandra offer applications a Paxos-based mechanism that ensures that replicas agree on a value as the outcome of a write or of concurrent writes, thereby establishing an order among concurrent writes.

Key characteristics of LWT:

  • Breaks LWW — instead of last-write-wins, the winner is determined by Paxos consensus
  • Prevents data loss and enforces uniqueness constraints
  • It is consensus per partition and per operation only — NOT cross-partitions, NOT for a sequence of operations (no multi-row transactions)
  • It is application-enforced, not database-enforced (unlike Postgres where constraints are automatic)
  • Applications that use LWT must NOT mix normal LWW writes and LWT writes on the same data — mixing corrupts the ordering guarantee
  • Write operations: add a conditional IF clause
  • Read operations: use consistency level SERIAL to perform Paxos reads

13. LWT — Enforcing Uniqueness (IF NOT EXISTS)

The classic use case for LWT: ensuring unique emails/usernames in a distributed database where no single-server uniqueness constraint exists.

The Problem Without LWT

In ScyllaDB without LWT, two clients can simultaneously INSERT rows with the same primary key. The last write wins (by timestamp). This means two users could "register" with the same username and one silently overwrites the other.

Solution: IF NOT EXISTS

-- Safe user registration
INSERT INTO users1 (u_id, activity_score, u_email, u_name) 
VALUES (12, 100.5, 'zurab@xx.ge', 'Zurab') IF NOT EXISTS;
-- Scenario 1: row with u_id=12 does NOT exist [applied] | u_id | activity_score | post_count | u_birthday | u_email | u_name -----------+------+----------------+------------+------------+---------+------- True | null | null | null | null | null | null (1 row) -- Result: INSERT applied. u_id=12, Zurab inserted. 
-- Attempt to insert duplicate (u_id=5 already has tamar@xx.ge)
INSERT INTO users1 (u_id, activity_score, u_email, u_name) 
VALUES (5, 100.5, 'zurab@xx.ge', 'Zurab') IF NOT EXISTS;
-- Scenario 2: row with u_id=5 ALREADY EXISTS [applied] | u_id | activity_score | post_count | u_birthday | u_email | u_name -----------+------+----------------+------------+------------+---------------+------- False | 5 | 100.5 | 10 | 2000-09-30 | tamar@xx.ge | tamar (1 row) -- Result: INSERT NOT applied. Existing row returned. Client must handle this.
LWT response always includes [applied] boolean:
- [applied] = True → operation succeeded
- [applied] = False → condition was not met; current row values are returned

The standard INSERT (without IF NOT EXISTS) overwrites existing data silently:

-- WITHOUT LWT — silently overwrites!
INSERT INTO users1 (u_id, activity_score, u_email, u_name) 
VALUES (5, 100.5, 'tamar@xx.ge', 'Tamar');
-- Before: u_id=5 had elene@example.com, Elene
-- After:  u_id=5 now has tamar@xx.ge, Tamar (data loss!)

14. LWT — Compare-And-Set (IF condition)

LWT also supports conditional updates: "only update IF the current value equals what I expect."

The Problem: Concurrent Read-Modify-Write

Two clients both read activity_score = 100.5 and want to update it:

Client 1 reads: activity_score = 100.5 Client 2 reads: activity_score = 100.5 Client 1 writes: UPDATE users1 SET activity_score = 500 WHERE u_id = 5; Client 2 writes: UPDATE users1 SET activity_score = 200 WHERE u_id = 5; Result with LWW: depends on timestamp. One write silently overwrites the other. Result should be: the first update wins, the second should only apply to the already-updated value.

Solution: IF condition (CAS)

-- Client 1: "set to 500, but only if current value is 100.5"
UPDATE users1 SET activity_score = 500 
WHERE u_id = 5 IF activity_score = 100.5;

-- Client 2: "set to 200, but only if current value is 100.5"
UPDATE users1 SET activity_score = 200 
WHERE u_id = 5 IF activity_score = 100.5;

With LWT (Paxos): exactly ONE of these will succeed (whichever wins the Paxos round). The other will get [applied] = False and the updated value (e.g., 500) back, so the client can decide what to do.

LWT protects the happened-before order and will not allow the overwrite of the earlier operation. It uses a Compare-And-Set (CAS) method. The condition is evaluated atomically with the write — no race condition.

CQL Syntax Summary for LWT

-- INSERT with IF NOT EXISTS
INSERT INTO table (pk, col) VALUES (val1, val2) IF NOT EXISTS;

-- UPDATE with condition
UPDATE table SET col = new_val WHERE pk = key IF col = expected_val;

-- DELETE with condition
DELETE FROM table WHERE pk = key IF col = expected_val;

-- Read with serial consistency (Paxos read — sees latest LWT-committed value)
CONSISTENCY SERIAL;
SELECT col FROM table WHERE pk = key;

15. Mixing LWW and LWT — Dangerous!

NEVER mix LWW (normal) writes and LWT writes on the same data partition!
If you use LWT for some writes and LWW for others, the ordering guarantee breaks completely.

Why Mixing is Dangerous — Example from Slides

-- Session 1 (LWW write):
session1: update users1 set activity_score = 500 where u_id = 5;

-- Session 2 (LWT write, concurrent):  
session2: update users1 set activity_score = 200 where u_id = 5 
          if activity_score = 100.5;

What happens?

  • Session 1 (LWW): sends write with timestamp T1, overwrites regardless of current value → score = 500
  • Session 2 (LWT/Paxos): reads current value via Paxos. But Session 1's LWW write may or may not be visible yet (depends on replication timing)
  • If Paxos reads score = 100.5 (before Session 1's LWW write is applied) → condition holds → sets score = 200 via Paxos
  • Then Session 1's LWW write (score = 500) arrives at replicas with timestamp T1
  • If T1 is higher than the Paxos write timestamp → LWW overwrites Paxos → score = 500 despite LWT "winning"

Result: Undefined, unpredictable behavior. The LWT guarantee is violated by the concurrent LWW write.

The rule: once you decide a partition uses LWT for ordering, ALL writes to that partition must go through LWT. Normal writes bypass Paxos and can silently overwrite LWT-committed values.

16. Scylla Paxos-Based LWT — Protocol Detail

ScyllaDB's LWT implementation uses a modified Paxos with 4 phases (the coordinator acts as the Paxos proposer):

The 4 Phases of LWT in ScyllaDB

Phase 1: Prepare and Promise

  1. Coordinator (proposer) generates a unique Paxos ballot = (timestamp, node_id)
  2. Coordinator sends PREPARE(ballot) to all RF replicas
  3. Each replica either:
    • Promises: "I will not accept any proposal with a lower ballot number." Returns its currently stored value.
    • Rejects: "I already promised a higher ballot." Returns the higher ballot it has promised + any currently accepted value.
  4. Coordinator waits for a majority quorum of promises
  5. If rejected by majority → coordinator generates a new higher ballot and retries from Phase 1
Coordinator (ballot=0b89ad92, ts=..., node=172.20.0.2) | |--PREPARE(0b89ad92)------> Replica 172.20.0.2 (local) |--PREPARE(0b89ad92)------> Replica 172.20.0.3 | |<--PROMISE(empty, empty)-- 172.20.0.3 |<--PROMISE(empty, empty)-- 172.20.0.2 | | "got enough replies to proceed" -- quorum reached

Phase 2: Read (check condition)

  1. Coordinator reads the current value from the quorum of replicas (to evaluate the IF condition)
  2. If the condition (IF activity_score = 100.5) is NOT met → coordinator aborts, returns [applied]=False
  3. If condition IS met → proceed to Phase 3

Phase 3: Propose (Accept)

  1. Coordinator sends ACCEPT(ballot, new_value) to all RF replicas
  2. Each replica accepts if the ballot is still the highest it has promised
  3. If a replica has in the meantime promised a higher ballot → it rejects
  4. Coordinator needs a quorum of ACCEPTED responses

Phase 4: Commit (Learn)

  1. Coordinator sends LEARN(new_value) to all replicas
  2. Replicas apply the new value to the base table (the actual data table)
  3. Paxos round complete
  4. Coordinator returns [applied]=True to the client
Performance implication: LWT requires 4 round trips (Prepare → Read → Propose → Commit) compared to 1 round trip for a normal write. This is why LWT is 4x slower and should only be used when the ordering guarantee is truly needed.

Ballot Format

Ballot = (timestamp, node_id). When two coordinators have the same timestamp, the node_id breaks the tie. The ballot ordering is total: (ts, n2) > (ts, n1) if n2 > n1. This ensures proposals from different coordinators have a strict order.

17. LWT Conflict Scenario 1

Setup: rf=3, node ordering n5>n4>n3>n2>n1; n1 and n2 are coordinators; n3, n4, n5 are replicas

n1 wants: UPDATE users SET score=100 WHERE u_id=1 IF score=50
n2 wants: UPDATE users SET score=0 WHERE u_id=1 IF score=50

n1's timeline:

  • Sends PREPARE(ts, n1) to n3, n4, n5. Same timestamp as n2, but (ts,n1) < (ts,n2) because n1 < n2
  • n3, n4 promise ballot (ts,n1), send their stored values (score=50)
  • n1 gets quorum of promises, checks condition: score=50=50 ✓ → condition holds
  • n1 sends ACCEPT(ts,n1, 100) to n3, n4, n5
  • Because of delay, n3 has NOT yet received n2's PREPARE ballot
  • n3 accepts (ts,n1,100). But n4 and n5 REJECT (ts,n1,100) because they already promised (ts,n2) which is higher
  • n1 does NOT get a quorum of accepts → FAILURE
  • n1 returns failure/timeout to client
  • (In pure Paxos: n1 generates new ballot (ts_higher,n1) and starts a new round)

n2's timeline (concurrent):

  • Sends PREPARE(ts, n2) to n3, n4, n5
  • n5 promises (ts,n2), sends score=50
  • n4 upgrades its promise to (ts,n2), sends score=50
  • n2 gets quorum, checks condition: score=50=50 ✓ → condition holds
  • n2 sends ACCEPT(ts,n2, 0) to n3, n4, n5
  • n4 and n5 accept (ts,n2,0) → quorum reached → DECISION DONE
  • n2 sends LEARN to n3, n4, n5; replicas apply score=0 to table
  • n2 returns [applied]=True to its client

Final result: score = 0 (n2 won)

18. LWT Conflict Scenario 2

Same setup. Key difference: n3 receives n2's PREPARE after it already accepted n1's proposal.

n1's timeline:

  • Sends PREPARE(ts,n1), n3 and n4 promise
  • n1 gets quorum, condition holds, sends ACCEPT(ts,n1, 100)
  • n3 accepts (ts,n1,100). n4 and n5 reject (already promised ts,n2)
  • n1 does NOT get quorum → fails

n2's timeline:

  • Sends PREPARE(ts,n2), n5 promises
  • n4 upgrades promise to (ts,n2)
  • n3 receives PREPARE(ts,n2) → upgrades promise to (ts,n2), sends back promise AND the accepted value: (ts,n1, 100)
  • n2 gets quorum promise. BUT n2 learns that n3 already accepted value 100 under ballot (ts,n1)
  • Paxos rule: n2 MUST propose the previously accepted value (100) instead of its own (0)
  • n2 sends ACCEPT(ts,n2, 100) — NOT 0!
  • n4 and n5 accept (ts,n2,100) → quorum reached
  • n2 sends LEARN, replicas apply score=100
  • n2 returns the result to client (needs testing — may return "applied" or "failure")

Final result: score = 100 (n1's value propagated through n2's round!)

Key Paxos rule demonstrated: If during Phase 1 you learn that some value was already accepted in a previous ballot, you MUST propose that value in Phase 2, not your own. This is how Paxos ensures consistency even when proposals race.

19. LWT Conflict Scenario 3

Same setup. n1 has some latency before sending ACCEPT.

n2's timeline (faster):

  • Sends PREPARE(ts,n2), n5 promises. n4 upgrades to (ts,n2)
  • n2 gets quorum, condition holds, sends ACCEPT(ts,n2, 0)
  • n4 and n5 accept (ts,n2,0) → quorum → DECISION DONE
  • n2 sends LEARN, replicas apply score=0
  • n2 returns [applied]=True

n1's timeline (slower):

  • Sends PREPARE(ts,n1), n3 and n4 promise initially
  • n1 gets quorum, condition holds (score still reads as 50 from Paxos read)
  • Sends ACCEPT(ts,n1, 100)
  • But n4 and n5 send back promised ballot (ts,n2) INCLUDING accepted value (ts,n2,0)
  • n1 cannot proceed — its ballot is lower
  • n1 returns failure/timeout to client

Final result: score = 0 (n2 won again)

20. LWT Conflict Scenario 4

rf=3, n3>n2>n1. Scenario where n1 completes first.

n1's timeline:

  • Sends PREPARE(ts,n1), n3 and n4 promise
  • n1 gets quorum, condition holds, sends ACCEPT(ts,n1, 100)
  • n3 and n4 accept → QUORUM REACHED
  • n1 sends LEARN to n3, n4, n5. Value 100 is committed. Paxos round complete.
  • n1 returns [applied]=True to client

n2's timeline (started concurrent, but n1 won first):

  • Sends PREPARE(ts,n2) to n3, n4, n5. n5 promises with score=50
  • n3 and n4 upgrade promise to (ts,n2) and send back: accepted value (ts,n1, 100)
  • n2: learns that value 100 is already chosen (n3 and n4 report accepted value)
  • n2 cannot continue with its own value 0 — it must acknowledge that 100 is already decided
  • n2 returns failure to client
  • (In pure Paxos but NOT LWT implementation: n2 would start a new Paxos round using value 100 to complete the commit)

Final result: score = 100 (n1 won)

Key pattern from all 4 scenarios: The Paxos ballot ordering (ts, node_id) determines which coordinator "wins" the prepare phase. But the critical safety rule is: if any coordinator learns during Phase 1 that a value was already accepted in a prior ballot, it MUST carry that value forward — ensuring no two Paxos rounds can commit different values.

21. LWT Error Handling

An error executing a conditional statement (LWT) does NOT necessarily mean it failed. This is a critical difference from normal writes!

In addition to standard failure causes (client-side timeouts, server overload, memory allocation errors), Paxos has its own scenarios leading to uncertainty:

  • Paxos livelock: Two coordinators keep outbidding each other's ballots. Neither reaches Phase 3. Solved in practice by exponential backoff + randomized retry delay.
  • Update started by one node, completed by another: If coordinator n1 starts a Paxos round but fails after Phase 3, another node (n2) may detect the incomplete round during its own Phase 1 and complete the commit — with n1's value, not n2's.
  • Timeout uncertainty: A timeout doesn't tell you whether the operation succeeded or not. The Paxos round may have completed on some replicas after the client timed out.
To know for sure whether an LWT operation succeeded after an error: a client MUST either:
  1. Read the value back (with CONSISTENCY SERIAL) to verify the current state, OR
  2. Retry the operation until it succeeds (if idempotent)

Why LWT Errors Are Uncertain (Paxos Takeover)

Example: n1 runs UPDATE...IF score=50 and gets a timeout after sending ACCEPT. Meanwhile, n2 starts its own LWT on the same partition. During n2's Phase 1, it reads n1's partially-completed Paxos state and sees n1's value was already accepted by some replicas. Paxos requires n2 to complete n1's proposal (with n1's value). So n1's update may have been applied even though n1 got a timeout.
