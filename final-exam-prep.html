Task 1 — Match Description to Database Type

What is this task asking?
You are given 6 real-world scenarios. For each one you must: (1) name the correct database type from the list, and (2) write 1-2 sentences explaining which specific feature of that type makes it the right fit. The four options are: Relational, Key-Value, Document, Wide-Column / Column-Family.
The exam tests whether you know the key characteristics of each type — not just names. Look for trigger words in each description:
Relational → "transactions", "referential integrity", "constraints", "ACID", "multiple tables in agreement"
Key-Value → "single identifier", "fetch as fast as possible", "no querying inside", "score per player"
Document → "flexible schema", "different attributes per item", "load all details at once", "evolving fields"
Wide-Column → "known query patterns", "availability over consistency", "partition key + sort key", "social feed / timeline"
Format your answer like this every time:
[Database Type]. Reason: [one specific characteristic that matches the scenario].
Question 1
A payment system where transferring funds must update several accounts together as one unit, every entry has to point to a valid account, and nothing committed may ever be lost.
Sample Solution: Relational. Reason: multi-record atomic transactions, referential integrity, and durability (ACID) are core strengths of RDBMS — exactly what a payment system needs.
Question 2
A user-session service for a busy website that stores each login token under a single identifier and needs to fetch it back as fast as possible, without ever looking inside it.
Sample Solution: Key-Value. Reason: put/get by exact key with no querying inside the value is the key-value model's core design, giving O(1) lookups at maximum speed.
Question 3
An online store catalog where different products carry different sets of attributes, new attributes appear regularly, and each product is normally loaded together with all its details at once.
Sample Solution: Document. Reason: varying attributes per record, flexible/dynamic schema, and loading a full record with all its data in one operation describe the document model perfectly.
Question 4
A social feed showing "items belonging to a given account, newest first," where query patterns are known beforehand and availability matters more than always reading the newest value.
Sample Solution: Wide-Column / Column-Family. Reason: query-first / denormalized design with a partition key (account) and clustering key (timestamp) serves known read patterns. Prioritizes availability over strong consistency (AP in CAP).
Question 5
A game leaderboard that tracks a score per player and needs quick increment-and-read operations on a player identifier.
Sample Solution: Key-Value. Reason: single-identifier access with atomic increment operations and no relational queries — exactly what key-value stores (e.g. Redis) are optimized for.
Question 6
A course-enrollment system that must stop two people from claiming the last available seat and the data about students, courses, and sign-ups has to stay in agreement.
Sample Solution: Relational. Reason: preventing the last-seat race requires transactional isolation (serializable/for update), and cross-table referential integrity keeps students, courses, and sign-ups consistent.

Task 2 — Scylla Consistency (Quorum)

What is this task asking?
This task tests your understanding of distributed consistency. You need to know: how ACID consistency differs from CAP consistency, why replicas can diverge and how they are reconciled, how the quorum formula W + R > N works, and how to compare two quorum configurations by availability and latency.
Always check W + R > N first before comparing configs. Then ask: how many node failures can reads survive? How many can writes survive? That tells you the tradeoff.
Key variables to memorize:
N = total replica count  |  W = replicas that must ack a write  |  R = replicas that must respond to a read
Read can tolerate N − R failures. Write can tolerate N − W failures.
Question 1
The word "consistency" means different things in ACID and in CAP. Define both.
Sample Solution: ACID consistency — a transaction never breaks the database's internal rules. Constraints, foreign keys, and data invariants stay correct. It is about keeping data valid inside one database.

CAP consistency — every node shows the same, most up-to-date value at the same time, so all clients see the same thing. It is about replicas agreeing with each other across a distributed system.
Question 2
Name and distinguish the two fundamentally different reasons two replicas can hold different values for the same object. Explain how the system resolves each.
Sample Solution: 1. Concurrent writes — different writes reached different coordinators with no happened-before relationship, so replicas genuinely hold conflicting values. Resolution requires a conflict-resolution rule (e.g. Last-Write-Wins, sibling values / application merge).

2. Replication lag — a write has been applied on some replicas but not yet propagated to others. There is a clear order, no conflict; it resolves on its own via eventual consistency (just needs time / read repair / anti-entropy).

Key distinction: case 1 needs a rule to decide which value wins; case 2 only needs time to converge.
Question 3
Explain how the quorum mechanism works in Scylla. State the roles of N, W, R and explain what the rule W + R > N guarantees and why.
Sample Solution: N = number of replicas that store a copy of the data.
W = how many replicas must confirm a write before it is considered successful.
R = how many replicas must respond to a read before a result is returned.

The coordinator sends the write/read to all N replicas, then waits for W (or R) acknowledgements. If it cannot collect them in time, it returns an error.

W + R > N guarantees that the write set and read set always overlap by at least one replica. That overlapping replica holds the latest write, so any read is guaranteed to see the most recent value — strong consistency.
Question 4
A cluster has N = 5. Compare configuration (a) W = 3, R = 3 with (b) W = 2, R = 4. Does each guarantee consistency? How do they differ in read vs write availability and latency, and which would generally be preferred? Justify.

Question 5
In consistency quorum, a write times out before a majority acknowledges and the coordinator returns failure to the client. Is the data guaranteed NOT to have been written? Explain.
Sample Solution: No. A timeout only means a quorum did not acknowledge in time. Some replicas may have already applied the write — and applied writes in Scylla are never rolled back. The write can still propagate to remaining replicas later via read repair and become fully persistent. The client must read the value back to confirm whether it was actually written.

Task 3 — Transaction Anomaly (Write Skew)

What is this task asking?
You are given a concrete hospital scenario with a business invariant (at least 1 doctor on call). You need to: (1) identify the anomaly, draw the interleaved schedule, and explain why the invariant breaks; (2) give two different prevention strategies and compare them.
The anomaly here is Write Skew — a special case that REPEATABLE READ does NOT prevent. Write skew happens when two transactions each read a shared condition, both decide it is safe to write, write to different rows, and together violate an invariant. Phantom reads are different (new rows appear). Write skew is about reading existing rows and writing to different ones.
Write skew is NOT prevented by REPEATABLE READ or READ COMMITTED. You need either SELECT FOR UPDATE (pessimistic locking) or SERIALIZABLE isolation (optimistic / SSI).
Question 1
Suppose Alice and Bob both try to go off-call at the same time. What problem/anomaly might arise? Illustrate it with a concrete schedule of the two transactions, and explain why the invariant ends up violated.
Sample Solution: Anomaly: Write Skew.

StepT1 (Alice)T2 (Bob)
1COUNT on_call → 2
2COUNT on_call → 2
3SET Alice off-call
4SET Bob off-call
5COMMITCOMMIT
Both transactions read count = 2, so each concludes it is safe to step down. They write to different rows, so neither blocks the other under REPEATABLE READ. Both commit. Final state: 0 doctors on call — invariant broken.

Each transaction was individually correct; only their combination violates the rule.
Question 2
Give two different ways to prevent this problem and compare them.
Sample Solution: Option A — SELECT … FOR UPDATE (pessimistic locking)
Change the count query to: SELECT * FROM doctors WHERE on_call = true FOR UPDATE
T1 locks all on-call rows. T2's identical query blocks until T1 commits. T2 then re-reads, sees count = 1, and refuses to step down.

Option B — SERIALIZABLE isolation level
Upgrade the isolation to SERIALIZABLE. The database detects that the two transactions cannot run serially without conflict and aborts one at commit time. The aborted transaction is retried — on retry it sees count = 1 and refuses.

Task 4 — Updates in Postgres (HOT vs COLD)

What is this task asking?
You must trace exactly how many physical locations Postgres writes to for each UPDATE, and explain why. The key concept is HOT (Heap-Only Tuple) updates vs regular (COLD) updates, and how indexes are affected. Always count: WAL + heap + affected indexes.
HOT update conditions (both must be true): (1) the changed column is not indexed, AND (2) the new row version fits on the same heap page. If either condition fails → COLD update → all indexes must be updated.
Postgres update = DELETE old row + INSERT new row (MVCC). Postgres never modifies a row in-place. The old tuple is marked dead, a new tuple is written on the heap. In a COLD update every index needs a new entry pointing to the new tuple's location.
Update 1
UPDATE product SET stock = 40 WHERE id = 1;
How many places does Postgres write to, and why?
Sample Solution: HOT update — 2 places.

Why HOT? stock is not indexed, and there is free space on the same heap page (given by the question). Both HOT conditions are satisfied.

Write locationReason
1. WAL (Write-Ahead Log)All changes must be logged first for durability / crash recovery.
2. Heap fileNew tuple version written on the same page; old tuple marked dead. No index entries need to change — the index still points to the same page and Postgres follows the HOT chain.
Update 2
UPDATE product SET price = 79 WHERE id = 1;
How many places does Postgres write to, and why?
Sample Solution: COLD update — 5 places.

Why COLD? price is indexed. Changing an indexed column means the index entry for the old tuple is stale — every index must now point to the new tuple's heap location. HOT is impossible.

Write locationReason
1. WALDurability log — always first.
2. Heap fileNew tuple version inserted (old marked dead).
3. Index on id (primary key)All indexes must add an entry pointing to the new tuple's location.
4. Index on nameSame reason — even though name didn't change, the row's physical location changed.
5. Index on pricePrice value changed, so the old index entry is wrong; new entry needed at new value.

Task 5 — State-based CRDT Set (OR-Set)

What is this task asking?
You must trace a Conflict-free Replicated Data Type (CRDT) OR-Set step by step, filling in each node's Add and Remove sets after each operation or merge. You then compute get(fruits) = elements in Add but not in Remove that were seen before the remove.
The two critical rules for this CRDT:
1. Merge = union — when nodes merge, their Add sets are unioned, their Remove sets are unioned.
2. Remove only works if the node has already seen the element — i.e. the element exists in that node's Add set at the time of removal. If not, the remove is silently discarded.
3. get(fruits) = All elements in any node's Add set minus those in any node's Remove set (at each merged state).
Concurrent add + remove of the same element: if the remover never saw the element (no merge in between), the add wins. This is the OR-Set (Observed-Remove Set) semantics.

Task 6 — Version Vectors

What is this task asking?
You trace two concurrent client writes through a 3-node cluster using version vectors. Each node maintains a vector clock (n1, n2, n3). You must fill in the Bob value and the updated version vector after each write propagates. Last-Write-Wins (LWW) is the conflict resolution strategy.
Version vector rules:
1. When a node receives a new write from a client, it increments its own slot in the vector.
2. When a write is replicated to another node, that node takes the max of each slot from both vectors (merge).
3. LWW = when two writes conflict (neither vector dominates), pick the one with the higher timestamp / latest write.
The exam note says: using LWW with version vectors defeats their purpose in real systems — if you always use LWW, you don't need version vectors. Version vectors are more meaningful with sibling values. Know this for theory questions.

Task 7 — MongoDB Read Concern

What is this task asking?
You need to define and distinguish two types of bad reads in MongoDB (dirty vs stale), and compare the two main readConcern levels. This is about MongoDB's replica set behaviour and consistency guarantees.
The core distinction to memorize:
Dirty read = data that could be rolled back (not yet majority-committed) — risk of seeing data that "never happened".
Stale read = data that is real and committed, but old — you're seeing an outdated version because replication lag.

readConcern: local → risk of dirty reads (returns latest on this node, not necessarily majority-committed).
readConcern: majority → risk of stale reads (only returns majority-committed data, which might be slightly behind the latest write).
Question 1
Explain what is a dirty read, and how can it occur in MongoDB.
Sample Solution: A dirty read returns a value that has not yet been acknowledged by a majority of nodes and could later be rolled back. In MongoDB, it occurs when you read a write that the primary has applied locally but has not yet reached a majority of secondaries. If that primary steps down before replication completes, MongoDB rolls back the write — so the data you read effectively never existed.
Question 2
Explain what is a stale read, and how can it occur in MongoDB.
Sample Solution: A stale read returns an old but real (already committed) value. It occurs because MongoDB replication is asynchronous: a secondary node may not yet have applied the most recent writes due to replication lag. Reading from a lagging secondary gives you an out-of-date value — the data is durable, but not current.
Question 3
Compare readConcern: local and readConcern: majority. Which one is more prone to stale reads? Explain.
Sample Solution:
readConcern: local readConcern: majority

readConcern: majority is more prone to stale reads. Because it only serves data that has been committed to a majority, it waits for the replication pipeline, so it always lags slightly behind the absolute latest write — trading freshness for safety against dirty reads.
